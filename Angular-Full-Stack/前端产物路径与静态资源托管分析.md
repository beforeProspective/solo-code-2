# Angular 全栈项目的前端产物路径与静态资源托管机制分析

## 一、生产环境：`__dirname` 与相对路径是如何让 Express 正确找到前端产物的？

在生产环境执行 `npm start`（即 `node dist/server/app.js`）时，Express 服务器通过两段信息拼接出前端编译产物在物理磁盘上的绝对路径。

### 1. 关键代码片段

`server/app.ts`（编译后为 `dist/server/app.js`）：

```typescript
const feDir = 'public/browser';
app.use('/', express.static(pathJoin(__dirname, `../${feDir}`)));
app.get('/*splat', (req, res) => {
  res.sendFile(pathJoin(__dirname, `../${feDir}/index.html`));
});
```

### 2. `__dirname` 的含义

Node.js 中的 `__dirname` 是一个特殊变量，**始终指向当前被执行的 JS 文件所在目录的绝对路径**。

- 入口文件：`E:\solo-code-2\Angular-Full-Stack\dist\server\app.js`
- 因此运行时 `__dirname` = `E:\solo-code-2\Angular-Full-Stack\dist\server`

### 3. 拼接过程（生产环境磁盘结构）

```
Angular-Full-Stack/
├── dist/
│   ├── public/            ← angular.json 中 outputPath = "dist/public"
│   │   └── browser/       ← @angular/build:application 自动追加的子目录
│   │       ├── index.html
│   │       ├── main.xxx.js
│   │       └── ...
│   └── server/            ← tsc -p server 的产物
│       └── app.js         ← 正在执行的文件 (__dirname 指向这里)
```

拼接演算：

```ts
pathJoin(__dirname, '../public/browser')
= pathJoin('E:\...\dist\server', '../public/browser')
= 'E:\...\dist\public\browser'
```

`express.static` 接收到的是**物理磁盘上的绝对路径**，因此 Express 可以精准地在该目录内按 URL 子路径寻找静态资源（`index.html`、`main.xxx.js`、资源文件等）。

SPA 的兜底路由 `app.get('/*splat')` 也是通过完全相同的拼接方式把 `index.html` 的绝对路径传给 `res.sendFile`。

### 4. 小结

- 运行 `node dist/server/app.js` 时，`__dirname` = `项目根/dist/server`。
- `feDir = 'public/browser'`，配合 `..` 向上一级，恰好回落到 `dist/public/browser`。
- 整个寻址链路成立的前提：`angular.json` 的 `outputPath` 与 `server/app.ts` 的 `feDir` 要严格匹配，且 Express 必须以编译后的产物文件（即 `dist/server/app.js`）作为入口。

## 二、开发环境下访问 3000 端口为什么会看到陈旧页面甚至报错？

### 1. `npm run dev` 的进程构成

`package.json` 中：

```json
"dev": "concurrently \"mongod\" \"ng serve --open\" \"tsc -w -p server\" \"nodemon dist/server/app.js\""
```

启动后会并行运行 4 个进程：

| 进程 | 作用 | 监听端口/目录 |
| --- | --- | --- |
| `mongod` | 数据库服务 | - |
| `ng serve --open` | Angular 开发服务器 | 4200（默认），内存编译、HMR、热更新 |
| `tsc -w -p server` | 后端 TS → JS 监听编译 | 写入 `dist/server/` |
| `nodemon dist/server/app.js` | 后端 API + 静态托管（Express） | 3000 |

### 2. 两个服务器的差异

| 端口 | 服务 | 前端资源来源 |
| --- | --- | --- |
| 4200 | `ng serve`（webpack-dev-server / Vite） | 内存中编译，随时反映最新代码，有 HMR |
| 3000 | `node dist/server/app.js` (Express) | 物理磁盘上的 `dist/public/browser`，只能读取**上次 `ng build` 的结果** |

### 3. 陈旧页面 & 报错的成因

- **陈旧页面**：在开发流程下，开发者通常只改 `client/` 源文件。`ng serve` 在 4200 端口的内存里直接构建最新代码，无需写入磁盘；但 3000 端口的 Express 只认 `dist/public/browser` 目录里的**磁盘文件**，而该目录只有在运行 `ng build`（或 `npm run build:dev`）时才会更新。所以访问 3000 端口拿到的是**上一次执行 `ng build` 时留下的历史快照**，自然是陈旧的。
- **路径找不到**：如果从未运行过 `ng build`，那么磁盘上根本就不存在 `dist/public/browser/index.html`。访问 `http://localhost:3000` 时：
  - `express.static` 找不到任何匹配文件；
  - 兜底路由 `res.sendFile(...)` 试图打开 `E:\...\dist\public\browser\index.html`，但文件在磁盘上缺失，于是 Node.js 抛出 `ENOENT: no such file or directory`，最终 Express 返回 500 或"找不到资源"类的系统错误。

### 4. 小结

- **4200 = 内存里的最新代码**（正确的开发入口）；
- **3000 = 磁盘上的历史快照**（仅供生产或构建后验证使用）；
- 开发时误访问 3000，要么看到旧页面，要么由于从未构建导致 `ENOENT` 报错。

## 三、若将 `outputPath` 改为 `dist/client`，`feDir` 应如何同步调整？

### 1. 新的磁盘产物结构

当 `angular.json` 中：

```json
"outputPath": "dist/client"
```

`@angular/build:application` 构建器会生成：

```
dist/
├── client/
│   └── browser/      ← 浏览器端产物
│       ├── index.html
│       └── ...
└── server/
    └── app.js
```

### 2. 保持同样的拼接规则

Express 端仍然以 `__dirname`（即 `dist/server`）为基准，向上一级（`..`）再进入 `client/browser`：

```ts
const feDir = 'client/browser';
app.use('/', express.static(pathJoin(__dirname, `../${feDir}`)));
app.get('/*splat', (req, res) => {
  res.sendFile(pathJoin(__dirname, `../${feDir}/index.html`));
});
```

### 3. 具体调整清单

| 文件 | 修改 |
| --- | --- |
| `angular.json` | `outputPath`: `"dist/public"` → `"dist/client"` |
| `server/app.ts` | `const feDir = 'public/browser';` → `const feDir = 'client/browser';` |
| `dist/server/app.js` | 通过 `tsc -p server` 自动重新编译，`feDir` 会从 TS 源文件同步过来 |

### 4. 额外提醒

- 若存在 `.gitignore`、Dockerfile、CI 脚本或部署脚本中硬编码了 `dist/public` 的路径，也需要一并替换为 `dist/client`，否则部署阶段仍会因为路径不匹配而找不到前端产物。
- 如果将来把 `@angular/build:application` 换成旧的 `@angular-devkit/build-angular:browser`，它不会自动生成 `browser` 子目录，届时 `feDir` 需改成 `'client'`（不再加 `/browser`）。因此 `feDir` 的真实含义应当是：**"构建器实际输出目录相对于 `dist/server` 的上一级路径"**。

---

## 四、结论

1. **生产环境寻址链路**：`__dirname`（`dist/server`）→ `../public/browser` → `dist/public/browser`，与 `angular.json` 的 `outputPath = "dist/public"` 完美对齐，因此 Express 能稳定地在磁盘上找到前端产物。
2. **开发端口差异**：4200 是 `ng serve` 的内存编译结果（最新），3000 是 Express 读取的磁盘快照（可能陈旧甚至不存在）。误访问 3000 会出现旧页面或 `ENOENT` 报错。
3. **路径重构的同步点**：只需把 `server/app.ts` 中的 `feDir` 从 `'public/browser'` 改为 `'client/browser'`，即可让 Express 在 `outputPath = "dist/client"` 下继续正确托管静态资源。
