# flatnotes 前端认证与路由防护机制分析

本文围绕 `client/router.js`、`client/api.js` 与 `client/tokenStorage.js` 三者的协同工作，回答三个问题：

1. 路由守卫如何调用 `authCheck` 核验登录状态，并避免无限重定向；
2. Axios 请求拦截器如何基于 `getStoredToken` 自动注入 `Authorization` 头部；
3. 401 响应时 `apiErrorHandler` 如何读取当前路由的 `fullPath` 并携带 `redirect` 参数跳转到登录页。

---

## 1. `router.beforeEach` 通过 `authCheck` 核验登录状态，并防止无限重定向循环

### 1.1 核心代码位置

路由守卫位于 [router.js#L44-L63](file:///e:/solo-code-2/flatnotes/client/router.js#L44-L63)：

```js
// Check the user is authenticated on first navigation (unless going to login)
let authChecked = false;
router.beforeEach(async (to) => {
  if (authChecked || to.name === "login") {
    return;
  }
  try {
    await authCheck();
    return;
  } catch (error) {
    if (error.response && error.response.status === 401) {
      return {
        name: "login",
        query: { [constants.params.redirect]: to.fullPath },
      };
    }
  } finally {
    authChecked = true;
  }
});
```

`authCheck` 函数定义于 [api.js#L68-L75](file:///e:/solo-code-2/flatnotes/client/api.js#L68-L75)：

```js
export async function authCheck() {
  try {
    const response = await api.get("api/auth-check");
    return response.data;
  } catch (response) {
    return Promise.reject(response);
  }
}
```

### 1.2 工作流程

1. **首次导航触发守卫**：全局变量 `authChecked` 初始为 `false`，因此每次进入应用的**第一次**路由跳转都会进入守卫主体。
2. **调用 `authCheck()`**：该函数通过 axios 实例 `api` 向 `api/auth-check` 发起 GET 请求。由于 `api/token` 是唯一会被请求拦截器跳过注入 token 的 URL（见第 2 节），`api/auth-check` 请求会自动带上 `Authorization: Bearer <token>`，服务端据此判断是否有效。
3. **成功放行**：`authCheck` 成功 `resolve`，守卫返回 `undefined`（即不进行重定向），导航继续到目标路由 `to`。
4. **失败（401）跳转登录页**：若 `authCheck` 抛错且状态码为 401，守卫返回一个路由描述对象 `{ name: "login", query: { redirect: to.fullPath } }`，Vue Router 据此跳转到登录页，并将当前目标路径作为 `redirect` 查询参数传递，便于登录成功后回跳。

### 1.3 防无限重定向的关键设计

无限重定向的典型场景是：守卫在登录页上又触发一次校验，从而再次跳转到登录页。flatnotes 通过两层机制彻底规避：

1. **对登录页直接放行**：`if (authChecked || to.name === "login") return;`
   当目标路由本身就是 `login` 时，守卫立即返回 `undefined`，不会再调用 `authCheck`，也不会再次 `return { name: "login", ... }`，从根本上切断 "登录页 → 登录页" 的循环。

2. **一次性的 `authChecked` 标志位**：
   `authChecked` 在 `finally` 块中被置为 `true`，无论 `authCheck` 成功或失败（401）都会执行。这意味着：
   - **首次导航后**：无论用户是否已登录，守卫后续所有导航都会直接 `return`，不再重复发送 `api/auth-check` 请求。
   - **避免重复网络请求**：一次页面生命周期中只请求一次鉴权接口。
   - **若服务端会话中途失效**：前端不再通过路由守卫主动发现，而是依赖第 3 节所述的响应拦截器（`apiErrorHandler`）在业务请求失败时再处理跳转。

换句话说，**路由守卫只在「应用初次加载时」做一次集中鉴权，后续的会话过期则由 API 层的响应拦截器处理**，两者分工清晰，从机制上避免了循环跳转。

## 2. Axios 请求拦截器基于 `getStoredToken` 注入 `Authorization` 头部

### 2.1 核心代码位置

请求拦截器位于 [api.js#L12-L26](file:///e:/solo-code-2/flatnotes/client/api.js#L12-L26)：

```js
api.interceptors.request.use(
  // If the request is not for the token endpoint, add the token to the headers.
  function (config) {
    if (config.url !== "api/token") {
      const token = getStoredToken();
      if (token) {
        config.headers.Authorization = `Bearer ${token}`;
      }
    }
    return config;
  },
  function (error) {
    return Promise.reject(error);
  },
);
```

`getStoredToken` 定义于 [tokenStorage.js#L21-L23](file:///e:/solo-code-2/flatnotes/client/tokenStorage.js#L21-L23)：

```js
export function getStoredToken() {
  return sessionStorage.getItem(tokenStorageKey);
}
```

其中 `tokenStorageKey` 为字符串 `"token"`（[tokenStorage.js#L1](file:///e:/solo-code-2/flatnotes/client/tokenStorage.js#L1)）。

### 2.2 令牌的生命周期

`tokenStorage.js` 提供了完整的凭证生命周期管理：

- **存储入口 `storeToken(token, persist)`**（[tokenStorage.js#L13-L19](file:///e:/solo-code-2/flatnotes/client/tokenStorage.js#L13-L19)）：
  - 立即写入 `sessionStorage`；
  - 若 `persist === true`，额外写入 `localStorage`（下次打开浏览器/新标签页可恢复）；
  - 同时写入 cookie（受 `SameSite=Strict` 保护）。
- **读取 `getStoredToken()`**：只从 `sessionStorage` 读取，保证当前会话级别的最新值。
- **恢复 `loadStoredToken()`**（[tokenStorage.js#L25-L30](file:///e:/solo-code-2/flatnotes/client/tokenStorage.js#L25-L30)）：应用启动时从 `localStorage` 迁移至 `sessionStorage`，实现"记住我"的跨标签页续期。
- **清除 `clearStoredToken()`**（[tokenStorage.js#L32-L37](file:///e:/solo-code-2/flatnotes/client/tokenStorage.js#L32-L37)）：登出时同时清理 `sessionStorage`、`localStorage` 和 cookie。

### 2.3 拦截器的注入流程

1. 业务代码（如 `getNotes`、`getNote`、`authCheck` 等）调用 `api.get(...)` / `api.post(...)` 时，Axios 先进入请求拦截器。
2. 拦截器判断 `config.url` 是否为 `api/token`：
   - **是 `api/token`**：这是登录换 token 的接口，尚未登录没有 token，直接跳过注入，避免把过期或无效 token 带过去造成混淆。
   - **不是 `api/token`**：调用 `getStoredToken()` 从 `sessionStorage` 取出 token；若存在，将 `config.headers.Authorization` 设置为 `Bearer <token>`。
3. 拦截器返回更新后的 `config`，Axios 继续发送携带 `Authorization` 头的请求。
4. 若 `sessionStorage` 中无 token（未登录），`getStoredToken()` 返回 `null`，则**不设置** `Authorization` 头，服务端会在鉴权失败时返回 401，交由第 3 节所述的响应处理逻辑统一跳转。

这种"每次请求都实时从 sessionStorage 读取"的设计，使得 `storeToken` / `clearStoredToken` 对 token 的修改能立即生效，无需重启或维护额外的内存副本。

## 3. `apiErrorHandler` 捕获当前路由的 `fullPath` 并携带 `redirect` 跳转到登录页

### 3.1 核心代码位置

[api.js#L28-L45](file:///e:/solo-code-2/flatnotes/client/api.js#L28-L45)：

```js
import router from "./router.js";

export function apiErrorHandler(error, toast) {
  if (error.response?.status === 401) {
    const redirectPath = router.currentRoute.value.fullPath;
    router.push({
      name: "login",
      query: { [constants.params.redirect]: redirectPath },
    });
  } else {
    console.error(error);
    toast.add(
      getToastOptions(
        "Unknown error communicating with the server. Please try again.",
        "Unknown Error",
        "error",
      ),
    );
  }
}
```

### 3.2 流程解析

1. **触发时机**：业务组件在调用 `api.js` 中的方法后，若服务端返回 401（token 缺失、过期或无效），Axios 响应拦截器未在此层统一处理，而是将错误抛出。业务代码在 `catch` 中调用 `apiErrorHandler(error, toast)`。

2. **读取当前路由的 `fullPath`**：
   ```js
   const redirectPath = router.currentRoute.value.fullPath;
   ```
   `router` 是从 [router.js](file:///e:/solo-code-2/flatnotes/client/router.js) 导入的 **Vue Router 实例**（"虚拟路由器实例"即此 `createRouter` 返回的对象）。在 Vue Router 4 中，`router.currentRoute` 是一个 `Ref`，因此通过 `.value` 访问当前路由对象，再取 `fullPath`（包含路径与查询串的完整 URL，如 `/note/hello?foo=bar`）。

3. **构造登录路由并跳转**：
   ```js
   router.push({
     name: "login",
     query: { [constants.params.redirect]: redirectPath },
   });
   ```
   使用**具名路由** `name: "login"` 而非硬编码路径，保证路由定义改动时跳转仍然有效。`constants.params.redirect` 的值为 `"redirect"`（[constants.js#L4](file:///e:/solo-code-2/flatnotes/client/constants.js#L4)），因此最终 URL 形如：

   ```
   /login?redirect=/note/hello%3Ffoo%3Dbar
   ```

4. **登录页接收并使用 `redirect`**：登录页路由在 [router.js#L16-L20](file:///e:/solo-code-2/flatnotes/client/router.js#L16-L20) 通过 `props: (route) => ({ redirect: route.query.redirect })` 将查询参数作为 prop 注入 `LogIn.vue`，登录成功后即可 `router.push(redirect || "/")` 回到原先页面。

### 3.3 与路由守卫的协同

- **初次加载**：由 `router.beforeEach` 调用 `authCheck` 集中完成鉴权；失败时同样以 `{ redirect: to.fullPath }` 跳转登录页，与 `apiErrorHandler` 保持一致的参数命名。
- **中途过期**：路由守卫因 `authChecked = true` 不再重复触发，这时**任何业务 API 请求返回 401** 都会进入 `apiErrorHandler`，再次把用户带到登录页。
- **避免循环**：由于登录页路由 `name === "login"` 在守卫中被直接放行，即使 `apiErrorHandler` 在任意时刻触发 `router.push({ name: "login", ... })`，都不会引发新的鉴权请求或二次重定向。

## 4. 三者协作总览

```
用户导航 / 业务请求
        │
        ▼
  router.beforeEach ──────► authCheck() ──► api.get("api/auth-check")
        │  (仅首次)                              │
        │                                        ▼
        │                              请求拦截器注入 Authorization
        │                              (getStoredToken → Bearer token)
        │                                        │
        │                                        ▼
        │                              服务端校验 ──► 200 / 401
        │                                        │
        │          ◄─────────────── 401 抛出 ────┘
        ▼
 返回 { name:"login", query:{ redirect: to.fullPath } }
        │
        ▼
  登录页 (LogIn.vue) 读取 redirect prop

— 其他 API 请求中途 401 ——► apiErrorHandler
                              │
                              ▼
                 router.currentRoute.value.fullPath
                              │
                              ▼
              router.push({ name:"login", query:{ redirect } })
```

整体上：
- `tokenStorage.js` 统一托管 token 的存、读、清、恢复；
- `api.js` 请求拦截器对除 `api/token` 外的所有请求统一携带 `Authorization`，响应失败时由 `apiErrorHandler` 捕获当前路由并携带 `redirect` 跳回登录页；
- `router.js` 的 `beforeEach` 在应用首次导航时通过 `authCheck` 集中鉴权一次，并用 `authChecked` 标志位与 `to.name === "login"` 的短路逻辑彻底避免重定向循环。
