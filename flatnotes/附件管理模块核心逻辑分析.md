# 附件管理模块核心逻辑分析

本文对 flatnotes 附件管理模块的三层核心逻辑进行分析：上传时对文件名的恶意字符防御、重名时的时间戳后缀回退策略，以及下载时 FastAPI 静态响应的流式分发。

---

## 1. 上传防御：`is_valid_filename` 的黑名单过滤

### 1.1 调用链

上传入口位于 [main.py](file:///e:/solo-code-2/flatnotes/server/main.py#L229-L245) 的 `post_attachment` 路由：

```python
@router.post("/api/attachments", dependencies=auth_deps,
             response_model=AttachmentCreateResponse)
def post_attachment(file: UploadFile):
    try:
        return attachment_storage.create(file)
    except ValueError:
        raise HTTPException(status_code=400,
                            detail=api_messages.invalid_attachment_filename)
    ...
```

实际的存储逻辑由 [FileSystemAttachments.create](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py#L25-L35) 承担，它在落盘前第一时间调用校验函数：

```python
def create(self, file: UploadFile) -> AttachmentCreateResponse:
    is_valid_filename(file.filename)   # ← 入口处即校验
    try:
        self._save_file(file)
    except FileExistsError:
        file.filename = self._datetime_suffix_filename(file.filename)
        self._save_file(file)
    return AttachmentCreateResponse(...)
```

### 1.2 `is_valid_filename` 的实现

校验函数位于 [helpers.py](file:///e:/solo-code-2/flatnotes/server/helpers.py#L16-L25)：

```python
def is_valid_filename(value):
    """Raise ValueError if the declared string contains any of the following
    characters: <>:"/\\|?*"""
    invalid_chars = r'<>:"/\|?*'
    if any(invalid_char in value for invalid_char in invalid_chars):
        raise ValueError(
            "title cannot include any of the following characters: "
            + invalid_chars
        )
    return value
```

### 1.3 黑名单范围与防御语义

黑名单覆盖的是 **Windows / 跨平台通用非法文件名字符集**，共 9 个字符：

| 字符 | 含义 | 防御目的 |
| --- | --- | --- |
| `<` `>` | 输入/输出重定向符 | 阻止 shell/命令注入式文件名 |
| `:` | NTFS 数据流分隔符、URL 协议分隔符 | 阻断 `C:\...`、`http:`、`file:` 等模式 |
| `"` | 双引号 | 阻断跨目录拼接与路径注入 |
| `/` | Unix/Linux 目录分隔符 | **最关键：直接阻断 `../` 路径穿越** |
| `\` | Windows 目录分隔符 | **阻断 `..\` 路径穿越与 UNC 路径 `\\server\share`** |
| `|` | 管道符 | 阻断 shell 管道注入 |
| `?` `*` | 通配符 | 阻止 glob 式多文件匹配攻击 |

### 1.4 对路径穿越（Path Traversal）的防御效果

- 所有包含 `..` 的穿越字符串必然包含 `/` 或 `\`，而二者已在黑名单中，因此像 `../../etc/passwd`、`..\\windows\\system32\\...` 等 payload 都会被 `ValueError` 直接拒绝。
- UNC 路径 `\\attacker\share\...` 由于 `\` 被禁用，无法进入存储目录。
- URL 伪协议 `file:///etc/passwd` 因包含 `:` 和 `/`，同样被拦截。

因此只要后续代码使用 `os.path.join(self.storage_path, filename)` 的方式拼接路径（见 [_save_file](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py#L45-L48)），**即使未做 `os.path.abspath`/`commonpath` 之类的二次归一化校验，也能依赖「黑名单 + 仅在指定目录内写入」的组合达到防御效果**。

## 2. 重名处理：`FileExistsError` + 时间戳后缀回退

### 2.1 重名检测的根源：`open(filepath, "xb")`

落盘逻辑在 [_save_file](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py#L45-L48) 中：

```python
def _save_file(self, file: UploadFile):
    filepath = os.path.join(self.storage_path, file.filename)
    with open(filepath, "xb") as f:
        shutil.copyfileobj(file.file, f)
```

关键点：

- `open` 使用的模式是 **`"xb"`**（`x` = exclusive creation，`b` = binary）。
- 这是 Python 3.3+ 引入的独占创建模式，**如果目标文件已存在，系统调用会直接抛出 `FileExistsError`**，而不会覆盖原有文件。
- 相比先 `os.path.exists` 再写入的 TOCTOU（检查与使用间的竞态）风险，`"x"` 模式在原子性上更安全。

### 2.2 `create` 方法的异常处理流程

回到 [create](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py#L25-L35)：

```python
def create(self, file: UploadFile) -> AttachmentCreateResponse:
    is_valid_filename(file.filename)
    try:
        self._save_file(file)                          # ① 首次尝试
    except FileExistsError:
        file.filename = self._datetime_suffix_filename(file.filename)  # ② 改名
        self._save_file(file)                          # ③ 再次尝试
    return AttachmentCreateResponse(
        filename=file.filename,
        url=self._url_for_filename(file.filename)
    )
```

流程解读：

1. 首次调用 `_save_file`，若文件已存在 → 内核抛出 `FileExistsError`。
2. 捕获异常后，调用 `_datetime_suffix_filename` 基于 **原始文件名** 派生新名（注意：是在修改 `file.filename` 对象属性之后再调用 `_save_file`，所以第二次写入用的是新名）。
3. 若新名仍然冲突（极小概率，同一秒内多次上传相同文件），第二次 `_save_file` 仍会抛出 `FileExistsError`，此时异常会冒泡到 `main.py` 的 `post_attachment`，后者会返回 HTTP 409 `attachment_exists`。

### 2.3 `_datetime_suffix_filename` 的实现

定义于 [file_system.py](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py#L50-L54)：

```python
def _datetime_suffix_filename(self, filename: str) -> str:
    timestamp = datetime.utcnow().strftime("%Y-%m-%dT%H-%M-%SZ")
    name, ext = os.path.splitext(filename)
    return f"{name}_{timestamp}{ext}"
```

- 使用 UTC 时间，避免时区差异。
- 时间戳格式 `%Y-%m-%dT%H-%M-%SZ` 采用 ISO 8601 风格，且把冒号替换为连字符，**避免触发 `is_valid_filename` 对 `:` 的黑名单**（否则二次校验会失败）。
- 通过 `os.path.splitext` 保留原始扩展名，避免把 `.pdf` 变成 `.pdf_2026-05-26...` 造成识别失败。

举例：上传 `report.pdf`，若已存在，则新名为 `report_2026-05-26T08-30-15Z.pdf`。

## 3. 下载分发：`FileResponse` 的流式静态响应

### 3.1 路由层

下载端点在 [main.py](file:///e:/solo-code-2/flatnotes/server/main.py#L199-L224) 定义了 **两个等价路径**：

```python
@router.get("/api/attachments/{filename}", dependencies=auth_deps)
@router.get("/attachments/{filename}",
            dependencies=auth_deps, include_in_schema=False)
def get_attachment(filename: str):
    try:
        return attachment_storage.get(filename)
    except ValueError:
        raise HTTPException(status_code=400,
                            detail=api_messages.invalid_attachment_filename)
    except FileNotFoundError:
        raise HTTPException(status_code=404,
                            detail=api_messages.attachment_not_found)
```

- `/api/attachments/{filename}`：标准 API 路径，出现在 OpenAPI schema。
- `/attachments/{filename}`：不进入 schema 的别名，用于在 Markdown / 富文本中构建直观的相对链接（如 `![pic](attachments/photo.jpg)`）。

两条路由共享同一个处理函数，认证依赖 `auth_deps`，下载行为受登录保护。

### 3.2 存储层：`FileSystemAttachments.get`

实现在 [file_system.py](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py#L37-L43)：

```python
def get(self, filename: str) -> FileResponse:
    is_valid_filename(filename)                              # ① 二次校验
    filepath = os.path.join(self.storage_path, filename)     # ② 路径拼接
    if not os.path.isfile(filepath):
        raise FileNotFoundError(f"'{filename}' not found.")  # ③ 存在性检查
    return FileResponse(filepath)                            # ④ 返回响应
```

要点：

1. **二次校验**：与上传一致，对 `filename` 再次调用 `is_valid_filename`，防御来自 URL 直接注入的恶意字符（如 `/attachments/../../etc/passwd`）。
2. **路径拼接**：`os.path.join(storage_path, filename)`。由于 `filename` 已不含路径分隔符，拼接结果必然是 `storage_path` 下的子文件，不会逃逸到父目录。
3. **存在性检查**：`os.path.isfile` 过滤目录路径与不存在的文件。
4. **返回 `FileResponse`**：FastAPI 的 `starlette.responses.FileResponse` 会接管后续的流式传输。

### 3.3 `FileResponse` 的流式分发机制

`FileResponse` 继承自 `starlette.responses.Response`，其关键行为包括：

- **分块读取（Streaming）**：内部使用 `anyio.open_file` 异步打开文件句柄，通过 `file_iterator` 逐块 yield 数据，避免把大文件一次性加载到内存。
- **正确的 `Content-Length`**：构造响应时调用 `os.stat` 获取文件大小，写入 `Content-Length` 头，使客户端能显示下载进度。
- **`Content-Type` 自动推断**：默认调用 `mimetypes.guess_type(filepath)` 根据扩展名返回 MIME 类型（如 `image/png`、`application/pdf`）；若未知则回退为 `application/octet-stream`。
- **条件请求支持**：Starlette 默认实现了 `ETag` / `Last-Modified`，配合 `If-Modified-Since`、`If-None-Match` 请求头可实现 304 Not Modified 缓存协商。
- **Range 请求支持**：对于大文件，`FileResponse` 会解析 `Range` 请求头，以 `206 Partial Content` 返回分块数据，便于断点续传。

因此在主路由中只需 `return FileResponse(filepath)`，**FastAPI / Starlette 会自动把磁盘上的静态附件文件以流式、带正确元信息的 HTTP 响应方式写回客户端**。

### 3.4 整体数据流示意

```
浏览器 GET /attachments/report.pdf
        │
        ▼
FastAPI 路由 get_attachment(filename)
        │  认证 auth_deps
        ▼
attachment_storage.get(filename)
        │  is_valid_filename(filename)   ← 防御路径穿越
        │  os.path.join(storage_path, filename)
        │  os.path.isfile(filepath)
        ▼
return FileResponse(filepath)
        │  Starlette 异步分块读取
        │  Content-Type / Content-Length / ETag
        ▼
HTTP 响应（200 / 206 / 304 / 404）
```

## 小结

| 子系统 | 关键函数/机制 | 作用 |
| --- | --- | --- |
| 上传防穿越 | `helpers.is_valid_filename` | 黑名单 `<>:"/\|?*`，从根源阻断 `../`、UNC、URL 伪协议 |
| 重名回退 | `open("xb")` → `FileExistsError` → `_datetime_suffix_filename` | 原子创建失败时以 ISO 时间戳改名重试，保留扩展名 |
| 下载分发 | `FileResponse`（Starlette） | 异步流式、自动 MIME、Range / 缓存协商 |

三层机制在 [file_system.py](file:///e:/solo-code-2/flatnotes/server/attachments/file_system/file_system.py) 中被统一封装，主路由 [main.py](file:///e:/solo-code-2/flatnotes/server/main.py#L199-L245) 只负责异常到 HTTP 状态码的映射，结构清晰、职责单一。
