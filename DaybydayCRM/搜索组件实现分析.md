# 搜索组件实现分析

## 一、前后端配合防止非法类型请求

### 三层防护机制

前端下拉类型切换、关键字输入阈值和后端模型白名单三者构成了三层防护，确保非法类型不会被发送到任意模型：

#### 1. 前端UI层限制（第一道防线）
在 [Search.vue](file:///e:/solo-code-2/DaybydayCRM/resources/assets/js/components/Search.vue#L10-L15) 的下拉菜单中，只提供四种合法类型选项：
- `clients`（客户）
- `tasks`（任务）
- `leads`（线索）
- `projects`（项目）

用户无法通过UI选择其他类型（如 `users` 或任意字符串）。

#### 2. 前端逻辑层限制（第二道防线）
- **输入阈值**：在 [Search.vue](file:///e:/solo-code-2/DaybydayCRM/resources/assets/js/components/Search.vue#L57-L60) 的 `search()` 方法中，当 `searchQueryLength < 3` 时直接返回空数组，不发送网络请求，减少无效请求。
- **类型受控**：`searchType` 仅通过 `setSearchType()` 方法赋值，值来自固定选项而非用户输入，确保请求URL的类型参数始终是预设值。

#### 3. 后端白名单验证（第三道防线）
在 [SearchController.php](file:///e:/solo-code-2/DaybydayCRM/app/Http/Controllers/SearchController.php#L17-L34) 中：
```php
$allowedTypes = [
    'client'   => Client::class,
    'clients'  => Client::class,
    'task'     => Task::class,
    'tasks'    => Task::class,
    'project'  => Project::class,
    'projects' => Project::class,
    'lead'     => Lead::class,
    'leads'    => Lead::class,
    'user'     => User::class,
    'users'    => User::class,
];
```

后端会：
1. 将类型转为小写后检查是否在白名单键中
2. 未通过验证时返回 `400 Invalid search type` 错误
3. 仅使用白名单中映射的类名实例化模型，防止任意类实例化漏洞

### 配合流程
```
用户选择下拉类型 → searchType 被设为合法值
用户输入关键字 → 达到3字符才发送请求
URL: /search/{query}/{searchType}
后端验证类型 → 在白名单中才查询对应模型
```

## 二、displayValue/searchLink 空值处理与设计考量

### 空值时的前端渲染

后端在 [SearchController.php](file:///e:/solo-code-2/DaybydayCRM/app/Http/Controllers/SearchController.php#L45-L47) 中有严格校验：
```php
if ( ! $class->displayValue() || ! $class->searchLink()) {
    continue;
}
```
如果任一方法返回空值，该条结果会被过滤掉，不会进入返回数组。

对前端的影响：
- 如果所有结果都被过滤，`results.length` 为0
- 前端会渲染 [Search.vue](file:///e:/solo-code-2/DaybydayCRM/resources/assets/js/components/Search.vue#L31-L33) 中的 "No Results" 提示
- 不会出现空链接或空白文本的列表项

### 为什么不能只返回原始数据库字段

这是一个**抽象层设计**，原因如下：

| 方法 | 作用 | 为什么不能用原始字段 |
|------|------|---------------------|
| `displayValue()` | 搜索结果的展示文本 | 1. Client 返回 `company_name`，Task/Lead/Project 返回 `title`，User 返回 `name`，各模型展示字段不同<br>2. 未来可扩展为组合字段（如"姓名(部门)"），无需修改前端 |
| `searchLink()` | 详情页跳转链接 | 1. 链接格式为 `/{type}/{external_id}`，需要拼接类型和外部ID，不是单一数据库字段<br>2. 由 [SearchableTrait.php](file:///e:/solo-code-2/DaybydayCRM/app/Traits/SearchableTrait.php#L51-L54) 统一提供，保证路由规范一致 |

如果直接返回原始数据库字段：
- 前端需要为每种模型写判断逻辑（用哪个字段显示、用哪个字段拼链接）
- 违反开闭原则，新增模型时需要修改前端代码
- 数据库字段变更会直接影响前端展示

## 三、切换全文检索时需保持不变的契约字段

当前LIKE查询返回的数据结构（后端拼装）：
```php
$result['hits']['hits'][] = [
    '_source' => [
        'display_value' => $class->displayValue(),
        'link'          => $class->searchLink()
    ]
];
```

前端在 [Search.vue](file:///e:/solo-code-2/DaybydayCRM/resources/assets/js/components/Search.vue#L22-L29) 中使用的字段：
```html
<li v-for="result in results">
    <a :href="result._source.link">
        <span>{{result._source.display_value}}</span>
    </a>
</li>
```

### 必须保持不变的字段

切换到 [SearchService.php](file:///e:/solo-code-2/DaybydayCRM/app/Services/Search/SearchService.php) 的Elasticsearch全文检索时，以下字段契约**必须保持不变**：

| 字段路径 | 用途 | 不变理由 |
|----------|------|----------|
| `response.data.hits.hits` | 结果数组外层结构 | 前端直接遍历此数组 |
| `result._source` | 单条结果的数据源包装 | Elasticsearch原生结构，前端用此路径访问 |
| `result._source.link` | 跳转链接 | `<a :href="...">` 直接绑定，改变会导致跳转失效 |
| `result._source.display_value` | 展示文本 | `{{...}}` 直接渲染，改变会导致显示为空或错误 |

### 现有兼容性

幸运的是，[SearchableTrait.php](file:///e:/solo-code-2/DaybydayCRM/app/Traits/SearchableTrait.php#L33-L49) 的 `toSearchArray()` 方法在索引数据到Elasticsearch时已经包含了这两个字段：
```php
$model['link']          = $this->searchLink();
$model['display_value'] = $this->displayValue();
```

因此Elasticsearch返回的 `_source` 中天然包含 `link` 和 `display_value`，切换时后端只需：
1. 调用 `SearchService::search()` 替代手动LIKE查询
2. 直接返回Elasticsearch结果（结构已兼容）
3. 前端代码无需任何修改

### 可扩展字段（不影响现有逻辑）
以下字段新增时不会破坏现有前端：
- `_score`（相关性得分）
- `highlight`（搜索高亮）
- 其他业务字段

只要不修改上述4个契约字段，跳转和展示逻辑就可以保持不变。
