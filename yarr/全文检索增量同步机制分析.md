# 全文检索增量同步机制分析

## 1. SyncSearch 函数中的索引同步逻辑与并发唯一性保障

### 1.1 为什么对 `search_rowid` 为 null 的行进行同步

在 [item.go](file:///e:/solo-code-2/yarr/src/storage/item.go#L382-L420) 的 `SyncSearch` 函数中，通过以下 SQL 查询待同步数据：

```sql
select id, title, content from items where search_rowid is null;
```

**设计原因：**

1. **增量同步标记**：`search_rowid` 字段作为同步状态标记，记录该条目在 FTS4 虚拟表中的 `rowid`。当值为 `null` 时表示该条目尚未被同步到全文检索表中。

2. **解耦设计**：文章抓取（`CreateItems`）与索引构建（`SyncSearch`）分离为两个独立流程。抓取时优先保证数据持久化，索引构建可异步延迟执行。

3. **幂等性**：每次同步只处理未同步的记录，避免重复索引和资源浪费。

### 1.2 并发插入相同文章的索引唯一性保障

**数据层面（items 表）的唯一性保障**：

在 [CreateItems](file:///e:/solo-code-2/yarr/src/storage/item.go#L121-L172) 函数中，通过数据库唯一约束防止重复文章：

```sql
insert into items (...) values (...)
on conflict (feed_id, guid) do update set last_arrived = :last_arrived
```

- 表结构定义了唯一索引：`create unique index if not exists idx_item_guid on items(feed_id, guid);`
- 同一 `feed_id` 下相同 `guid` 的文章只会存在一条记录

**索引层面（search 表）的并发问题分析**：

当前实现存在**潜在的并发重复索引风险**：

1. `SyncSearch` 函数没有开启事务，两步操作（插入 search 表 → 更新 items 表）非原子执行
2. 如果多个 goroutine 同时调用 `SyncSearch`，可能出现：
   - 线程 A 和 B 同时查询到同一条 `search_rowid is null` 的记录
   - 线程 A 先插入 search 表，获得 rowid=N
   - 线程 B 也插入 search 表，获得 rowid=M（同内容的重复索引）
   - 线程 A 更新 `search_rowid=N`，线程 B 更新 `search_rowid=M`（覆盖）
   - 最终 search 表中存在两条索引，但只有 rowid=M 被引用，rowid=N 成为悬空数据

**SQLite 层面的缓解**：

SQLite 采用**单写多读**模型（WAL 模式下也是单写），写入操作本质串行化。但在应用层并发调用时，仍可能出现查询结果重叠导致重复插入。

## 2. 纯文本提取的作用与噪音防护

### 2.1 `htmlutil.ExtractText` 的实现机制

在 [utils.go](file:///e:/solo-code-2/yarr/src/content/htmlutil/utils.go#L48-L64) 中，`ExtractText` 函数实现：

```go
func ExtractText(content string) string {
    tokenizer := html.NewTokenizer(strings.NewReader(content))
    buffer := bytes.Buffer{}
    for {
        token := tokenizer.Next()
        if token == html.ErrorToken {
            break
        }
        if token == html.TextToken {
            buffer.WriteString(html.UnescapeString(string(tokenizer.Text())))
        }
    }
    text := buffer.String()
    text = strings.TrimSpace(text)
    text = whitespaceRegex.ReplaceAllLiteralString(text, " ")
    return text
}
```

### 2.2 防止的全文检索噪音类型

| 噪音类型 | 具体示例 | 影响 |
|---------|---------|------|
| **HTML 标签** | `<div>`, `<p class="article">`, `<script>` | 标签名、CSS 类名会被误索引为搜索关键词 |
| **HTML 属性** | `class="content-wrapper"`, `id="main"` | 属性值可能包含大量非内容字符串 |
| **HTML 实体** | `&amp;`, `&lt;`, `&nbsp;` | 未解码的实体会导致搜索匹配失败 |
| **多余空白** | 连续空格、换行符、制表符 | 产生无意义的 token，影响分词质量 |
| **内嵌脚本** | JavaScript 代码块 | 代码变量名、关键字会污染索引 |
| **样式代码** | CSS 样式内容 | 选择器、属性名被错误索引 |

### 2.3 全文检索质量提升

通过纯文本提取，确保 FTS4 索引的 `content` 字段只包含：
- 用户实际可阅读的文章正文内容
- 规范化的空白字符（单空格分隔）
- 解码后的正常文本字符

这使得搜索匹配更加精准，避免搜索"div"、"class"等技术词汇时返回不相关结果。

## 3. 断点续传与悬空数据处理机制

### 3.1 断点续传实现原理

`SyncSearch` 采用**逐条处理+即时标记**的方式实现断点续传：

```go
for _, item := range items {
    // 1. 插入 search 虚拟表
    result, err := s.db.Exec(`insert into search ...`)
    
    // 2. 插入成功后立即更新标记
    if numrows, err := result.RowsAffected(); err == nil && numrows == 1 {
        if rowId, err := result.LastInsertId(); err == nil {
            s.db.Exec(`update items set search_rowid = ? where id = ?`, rowId, item.Id)
        }
    }
}
```

**恢复机制**：
- 每条记录处理完成后立即更新 `search_rowid`
- 若中途崩溃，已处理记录已有标记，未处理记录仍为 `null`
- 下次执行时自动从第一条 `search_rowid is null` 的记录继续

### 3.2 悬空数据问题分析

**悬空数据定义**：已成功插入 `search` 虚拟表，但 `items.search_rowid` 尚未更新的记录。

**产生场景**：
1. 插入 search 表成功（步骤 1 完成）
2. 更新 items 表失败或程序崩溃（步骤 2 未执行）
3. 此时 search 表中的新记录没有任何 items 记录引用它

**当前代码的处理机制**：

❌ **没有主动清理机制**

1. 悬空数据会一直存在于 `search` 表中，占用磁盘空间
2. 由于搜索查询通过 `items.search_rowid` 关联，悬空数据永远不会被命中
3. `del_item_search` 触发器只在 items 记录删除时触发，无法清理悬空数据

**潜在的覆写/清理方式**：

1. **重新同步时的重复插入**：下次 `SyncSearch` 会再次处理该条 items 记录，在 search 表中插入第二条重复索引（形成新的悬空数据风险）。

2. **DELETE 触发器的局限性**：
   ```sql
   create trigger if not exists del_item_search after delete on items begin
     delete from search where rowid = old.search_rowid;
   end;
   ```
   该触发器只能清理已知关联的索引，无法发现悬空数据。

### 3.3 悬空数据的清理建议（当前未实现）

可通过定期执行以下 SQL 清理悬空索引：

```sql
delete from search where rowid not in (
    select search_rowid from items where search_rowid is not null
);
```

或在 `SyncSearch` 开始前增加前置清理步骤。

## 总结

| 问题 | 结论 |
|-----|------|
| **增量同步标记** | `search_rowid is null` 作为待同步标记，实现解耦的增量同步 |
| **并发唯一性** | items 表有唯一约束保障；search 表有潜在重复风险，依赖 SQLite 单写模型缓解 |
| **纯文本提取** | 防止 HTML 标签、属性、脚本等技术噪音污染全文索引 |
| **断点续传** | 逐条处理即时标记，崩溃后自动从断点恢复 |
| **悬空数据** | 无主动清理机制，存在 search 表空间膨胀风险 |
