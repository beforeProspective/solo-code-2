# VBlog 分布式定时任务集群安全方案分析

## 一、业务异常分析

### 1. 问题现象
在不修改当前代码的情况下，将 VBlog 服务水平扩展为 3 个集群节点后，**每天凌晨的 PV 统计数据会被重复插入 3 次**，导致 PV 统计结果严重失真（实际 PV 值 ≈ 真实值 × 3）。

### 2. 底层原因分析

#### 2.1 当前代码实现
定时任务定义在 [DataStatisticsComponent.java](file:///e:/solo-code-2/VBlog/blogserver/src/main/java/org/sang/service/DataStatisticsComponent.java#L15-L19) 中：

```java
// 每天执行一次，统计PV
@Scheduled(cron = "1 0 0 * * ?")
public void pvStatisticsPerDay() {
    articleService.pvStatisticsPerDay();
}
```

最终执行的 SQL 在 [ArticleMapper.xml](file:///e:/solo-code-2/VBlog/blogserver/src/main/java/org/sang/mapper/ArticleMapper.xml#L101-L103) 中：

```sql
INSERT INTO pv(countDate, pv, uid) 
SELECT CURRENT_DATE(), totalPv-pv, t.`uid` 
FROM pvview p, totalpvview t 
WHERE p.`uid`=t.`uid`
```

#### 2.2 根本原因
Spring Framework 的 `@Scheduled` 注解是**单机调度机制**，不具备分布式协调能力：

| 问题点 | 具体说明 |
|--------|----------|
| **无分布式锁** | 每个节点的 Spring 容器独立加载并触发定时任务，节点间无任何协调机制 |
| **执行时间一致** | 3 个节点会在每天 00:00:01 几乎同时触发相同的 INSERT 语句 |
| **SQL 非幂等** | 普通 INSERT 语句不具备去重能力，每次执行都会新增一条记录 |
| **无唯一约束** | pv 表的 `(countDate, uid)` 组合没有唯一索引，允许重复数据存在 |

#### 2.3 异常流程示意

```
时间轴: 00:00:01
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  节点 A     │────▶│ 执行 INSERT │────▶│  插入成功   │
└─────────────┘     └─────────────┘     └─────────────┘
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  节点 B     │────▶│ 执行 INSERT │────▶│  插入成功   │  ← 重复数据1
└─────────────┘     └─────────────┘     └─────────────┘
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  节点 C     │────▶│ 执行 INSERT │────▶│  插入成功   │  ← 重复数据2
└─────────────┘     └─────────────┘     └─────────────┘
```

## 二、跨节点解决方案

以下提出 **4 种** 集群安全的改造方案，按实施复杂度从低到高排列。

---

### 方案一：数据库唯一索引 + 幂等性设计（推荐，改动最小）

#### 核心思路
利用数据库的**唯一约束**保证数据不重复，将 SQL 改造为**幂等操作**。

#### 实施步骤

**步骤 1：添加唯一索引**
```sql
ALTER TABLE pv ADD UNIQUE INDEX uk_countDate_uid (countDate, uid);
```

**步骤 2：修改 SQL 语句**
将 [ArticleMapper.xml](file:///e:/solo-code-2/VBlog/blogserver/src/main/java/org/sang/mapper/ArticleMapper.xml#L101-L103) 中的 INSERT 改为：

```xml
<insert id="pvStatisticsPerDay">
    INSERT IGNORE INTO pv(countDate, pv, uid) 
    SELECT CURRENT_DATE(), totalPv-pv, t.`uid` 
    FROM pvview p, totalpvview t 
    WHERE p.`uid`=t.`uid`
</insert>
```

或者使用 `ON DUPLICATE KEY UPDATE`（如果需要覆盖更新）：

```xml
<insert id="pvStatisticsPerDay">
    INSERT INTO pv(countDate, pv, uid) 
    SELECT CURRENT_DATE(), totalPv-pv, t.`uid` 
    FROM pvview p, totalpvview t 
    WHERE p.`uid`=t.`uid`
    ON DUPLICATE KEY UPDATE pv = VALUES(pv)
</insert>
```

#### 优缺点
| 优点 | 缺点 |
|------|------|
| ✅ 代码改动极小，仅需修改 SQL | ❌ 3 个节点仍会同时执行 SQL，造成数据库资源浪费 |
| ✅ 不引入任何新的中间件 | ❌ 无法扩展到更复杂的调度场景 |
| ✅ 数据一致性由数据库保证 | ❌ 唯一索引可能影响 INSERT 性能（可接受） |
| ✅ 天然支持节点故障转移 | |

---

### 方案二：Redis 分布式锁（推荐，常用方案）

#### 核心思路
使用 Redis 的 `SETNX`（Set if Not Exists）机制实现分布式锁，**只有获取到锁的节点才能执行定时任务**。

#### 实施步骤

**步骤 1：引入 Redis 依赖（如未引入）**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
```

**步骤 2：修改定时任务逻辑**

```java
@Component
public class DataStatisticsComponent {
    
    @Autowired
    private ArticleService articleService;
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String LOCK_KEY = "pv:statistics:lock";
    private static final long LOCK_EXPIRE = 300; // 锁过期时间5分钟

    @Scheduled(cron = "1 0 0 * * ?")
    public void pvStatisticsPerDay() {
        Boolean lock = redisTemplate.opsForValue()
            .setIfAbsent(LOCK_KEY, "locked", LOCK_EXPIRE, TimeUnit.SECONDS);
        
        if (Boolean.TRUE.equals(lock)) {
            try {
                articleService.pvStatisticsPerDay();
            } finally {
                redisTemplate.delete(LOCK_KEY);
            }
        }
    }
}
```

#### 优化建议：使用 Redisson 框架
```java
@Scheduled(cron = "1 0 0 * * ?")
public void pvStatisticsPerDay() {
    RLock lock = redissonClient.getLock(LOCK_KEY);
    try {
        boolean locked = lock.tryLock(0, LOCK_EXPIRE, TimeUnit.SECONDS);
        if (locked) {
            articleService.pvStatisticsPerDay();
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

#### 优缺点
| 优点 | 缺点 |
|------|------|
| ✅ 只有一个节点执行任务，节省资源 | ❌ 需要额外引入 Redis（如果项目已有则无此问题） |
| ✅ 性能较好，Redis 操作快 | ❌ 需注意锁的过期时间和释放逻辑 |
| ✅ 支持更复杂的分布式场景 | ❌ Redis 单点故障可能导致锁失效（可使用 Redis 集群） |
| ✅ 社区方案成熟，Redisson 提供完善的锁实现 | |

### 方案三：数据库乐观锁/调度状态表

#### 核心思路
创建一张 `schedule_task` 表记录任务执行状态，利用数据库的行级锁保证只有一个节点能"抢占"到执行权。

#### 实施步骤

**步骤 1：创建任务调度状态表**
```sql
CREATE TABLE schedule_task (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    task_name VARCHAR(64) NOT NULL COMMENT '任务名称',
    execute_date DATE NOT NULL COMMENT '执行日期',
    status TINYINT DEFAULT 0 COMMENT '0-待执行, 1-执行中, 2-已完成',
    lock_expire_time DATETIME COMMENT '锁过期时间',
    UNIQUE KEY uk_task_date (task_name, execute_date)
);

-- 初始化任务记录（每天0点由前一天的数据自动生成，或提前批量生成）
INSERT INTO schedule_task (task_name, execute_date, status) 
VALUES ('pvStatistics', CURRENT_DATE(), 0)
ON DUPLICATE KEY UPDATE status = status;
```

**步骤 2：修改定时任务逻辑**
```java
@Scheduled(cron = "1 0 0 * * ?")
@Transactional
public void pvStatisticsPerDay() {
    // 1. 尝试抢占任务（使用行级锁）
    int updated = articleMapper.tryLockTask(
        "pvStatistics", 
        LocalDate.now(), 
        LocalDateTime.now().plusMinutes(5)
    );
    
    if (updated > 0) {
        try {
            // 2. 抢到锁，执行统计
            articleService.pvStatisticsPerDay();
            
            // 3. 标记任务完成
            articleMapper.markTaskCompleted("pvStatistics", LocalDate.now());
        } catch (Exception e) {
            // 异常处理，可选择重置状态
            articleMapper.resetTaskStatus("pvStatistics", LocalDate.now());
            throw e;
        }
    }
}
```

**步骤 3：添加 Mapper 方法**
```xml
<update id="tryLockTask">
    UPDATE schedule_task 
    SET status = 1, lock_expire_time = #{expireTime}
    WHERE task_name = #{taskName} 
      AND execute_date = #{executeDate} 
      AND status = 0
    LIMIT 1
</update>
```

#### 优缺点
| 优点 | 缺点 |
|------|------|
| ✅ 不引入新中间件，利用现有数据库 | ❌ 增加了数据库表和业务逻辑复杂度 |
| ✅ 支持任务状态追踪和审计 | ❌ 需要定期清理或生成任务记录 |
| ✅ 可扩展支持任务失败重试 | ❌ 数据库成为单点（但一般数据库已有高可用） |

### 方案四：使用分布式任务调度框架（XXL-Job / Elastic-Job / Quartz）

#### 核心思路
将定时任务从应用中剥离，交由专业的**分布式任务调度平台**统一管理。

#### 以 XXL-Job 为例的改造方案

**步骤 1：引入 XXL-Job 依赖**
```xml
<dependency>
    <groupId>com.xuxueli</groupId>
    <artifactId>xxl-job-core</artifactId>
    <version>2.4.0</version>
</dependency>
```

**步骤 2：配置 XXL-Job 执行器**
```properties
xxl.job.admin.addresses=http://xxl-job-admin:8080/xxl-job-admin
xxl.job.executor.appname=vblog-executor
xxl.job.executor.port=9999
```

**步骤 3：改造定时任务**
```java
@Component
public class PvStatisticsJob {
    
    @Autowired
    private ArticleService articleService;
    
    @XxlJob("pvStatisticsPerDayHandler")
    public void pvStatisticsPerDay() {
        // 业务逻辑不变
        articleService.pvStatisticsPerDay();
    }
}
```

**步骤 4：在 XXL-Job 管理后台配置任务**
- 任务配置：`Cron = 1 0 0 * * ?`
- 路由策略：选择 **"第一个"** 或 **"随机"**（保证只有一个节点执行）
- 高级配置：可设置任务超时、失败重试等

#### 优缺点
| 优点 | 缺点 |
|------|------|
| ✅ 专业的分布式调度能力，功能完善 | ❌ 需要额外部署和维护调度中心 |
| ✅ 支持任务可视化管理、监控告警 | ❌ 引入了新的技术栈和运维成本 |
| ✅ 支持动态调整 cron、手动触发 | ❌ 对于单个简单任务来说"过重" |
| ✅ 支持任务分片、广播等复杂场景 | |

## 三、方案对比与推荐

| 方案 | 改造成本 | 性能 | 可靠性 | 适用场景 | 推荐指数 |
|------|----------|------|--------|----------|----------|
| **方案一：唯一索引** | 极低 ⭐ | 中 | 高 | 简单插入场景，已有数据库 | ⭐⭐⭐⭐ |
| **方案二：Redis 锁** | 低 ⭐⭐ | 高 | 中高 | 已有 Redis，追求性能 | ⭐⭐⭐⭐⭐ |
| **方案三：数据库状态表** | 中 ⭐⭐⭐ | 中 | 高 | 不引入新中间件，需任务追踪 | ⭐⭐⭐ |
| **方案四：XXL-Job 等框架** | 高 ⭐⭐⭐⭐ | 高 | 很高 | 任务数量多，需要管理平台 | ⭐⭐⭐⭐ |

### 最终推荐

1. **短期快速修复**：采用 **方案一（唯一索引 + INSERT IGNORE）**，2 行代码即可解决问题，风险最低。

2. **长期优雅方案**：采用 **方案二（Redis 分布式锁）**，如果项目中已有 Redis，这是最均衡的选择。

3. **未来规划**：如果后续会有更多定时任务，建议直接上 **方案四（XXL-Job）**，一劳永逸解决所有分布式调度问题。

## 四、注意事项

无论采用哪种方案，都需要考虑以下问题：

1. **任务幂等性**：确保任务重复执行不会产生副作用（这是分布式系统的基本要求）

2. **异常处理**：任务执行失败时的告警和重试机制

3. **执行超时**：设置合理的锁过期时间，防止节点宕机导致"死锁"

4. **时钟同步**：集群节点间需要进行时钟同步（NTP），避免 cron 触发时间差异过大

5. **数据库事务**：确保统计操作在同一个事务中完成，避免部分执行
