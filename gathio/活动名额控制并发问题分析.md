# 活动名额控制与RSVP并发问题深度分析

## 一、并发提交场景下的超卖风险分析

### 1.1 当前实现流程

当前系统在三个关键入口处理RSVP操作，均采用"**读取-修改-写入**"（Read-Modify-Write）的非原子模式：

| 入口 | 文件位置 | 主要逻辑 |
|------|---------|---------|
| Web预注册 | [routes.js:466-532](file:///e:/solo-code-2/gathio/src/routes.js#L466-L532) | `findOne`读取 → push新attendee → `save`保存 |
| Web确认参加 | [event.ts:789-930](file:///e:/solo-code-2/gathio/src/routes/event.ts#L789-L930) | `findOne`读取 → 容量检查 → 修改attendee → `save`保存 |
| ActivityPub | [activitypub.ts:173-191](file:///e:/solo-code-2/gathio/src/lib/activitypub.ts#L173-L191) | `findOne`读取 → 容量检查 → push新attendee → `save`保存 |

核心容量校验代码（[event.ts:816-821](file:///e:/solo-code-2/gathio/src/routes/event.ts#L816-L821)）：

### 1.2 超卖风险场景

假设活动上限为100人，当前已批准99人，剩余1个名额。此时有3个用户在**同一毫秒**并发提交：

```
时间轴（毫秒级）：
T0:  请求A → findOne → 读取到已批准=99 → freeSpots=1 ✓ 通过
T0:  请求B → findOne → 读取到已批准=99 → freeSpots=1 ✓ 通过
T0:  请求C → findOne → 读取到已批准=99 → freeSpots=1 ✓ 通过
T1:  请求A → 修改attendee.approved=true → save() → 已批准=100
T1:  请求B → 修改attendee.approved=true → save() → 已批准=101 ❌ 超卖
T1:  请求C → 修改attendee.approved=true → save() → 已批准=102 ❌ 超卖
```

**问题本质**：读取（`findOne`）和写入（`save`）是两个独立的异步操作，之间没有任何锁或原子性保证。多个请求可以读取到相同的"旧状态"，各自完成校验后并发写入，导致最终人数超过上限。

### 1.3 实际影响范围

- **无需审核的活动**：风险最高。用户提交即自动批准，直接占用名额，极可能超卖。
- **需要审核的活动**：风险相对较低但仍然存在。审核阶段的并发批准操作同样存在此问题。
- **ActivityPub联邦用户**：跨实例RSVP同样受此影响，且网络延迟可能加剧并发冲突。

## 二、多实例集群部署下的排他性问题与解决方案

### 2.1 为何findOne + save无法保证排他性

Mongoose的`findOne()` + `save()`模式在单进程单实例场景下，由于Node.js事件循环的单线程特性，可能偶然避免冲突。但在**多实例集群部署**下，这种模式完全失效，原因如下：

1. **内存状态隔离**：每个Node.js实例独立维护从MongoDB读取的文档副本，实例之间无共享内存和协调机制。

2. **无数据库锁**：MongoDB默认使用MVCC（多版本并发控制），但普通的`findOne`查询不会对文档加锁。

3. **写入时不验证版本**：Mongoose的`save()`默认使用`_id`匹配文档，不会检查读取后的文档是否被其他进程修改。

```
集群部署示意：

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Node实例A  │     │  Node实例B  │     │  Node实例C  │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           ▼
                    ┌─────────────┐
                    │  MongoDB   │
                    └─────────────┘

问题：三个实例各自findOne，各自修改，各自save，MongoDB无法感知冲突。
```

### 2.2 MongoDB原子更新解决方案

应使用MongoDB的原子更新指令替代"读取-修改-写入"模式。以下是几种可行方案：

#### 方案A：findOneAndUpdate + 条件查询（推荐）

将容量校验逻辑嵌入查询条件，让MongoDB在原子操作中完成"检查+更新"：

```typescript
// 原子更新：只有当已批准人数 < maxAttendees时，才添加/更新attendee
const result = await Event.findOneAndUpdate(
  {
    id: eventID,
    $expr: {
      $lt: [
        {
          $sum: {
            $map: {
              input: "$attendees",
              in: {
                $cond: [
                  { $and: [
                    { $eq: ["$$this.status", "attending"] },
                    { $or: [
                      { $eq: ["$approveRegistrations", false] },
                      { $eq: ["$$this.approved", true] }
                    ]}
                  ]},
                  { $ifNull: ["$$this.number", 1] },
                  0
                ]
              }
            }
          }
        },
        "$maxAttendees"
      ]
    }
  },
  {
    $set: { "attendees.$[elem].status": "attending", "attendees.$[elem].approved": true },
  },
  {
    arrayFilters: [{ "elem.removalPassword": removalPassword }],
    new: true
  }
);

if (!result) {
  // 更新失败，说明名额已满
  return res.status(403).json({ error: "Not enough spots available." });
}
```

#### 方案B：使用乐观并发控制（版本号）

在Schema中添加`versionKey`或自定义`__v`字段，更新时校验版本：

```typescript
const event = await Event.findOne({ id: eventID });
const currentVersion = event.__v;

// 修改attendee...

const result = await Event.findOneAndUpdate(
  { id: eventID, __v: currentVersion },  // 版本必须匹配
  { $set: { attendees: event.attendees }, $inc: { __v: 1 } },
  { new: true }
);

if (!result) {
  // 版本不匹配，说明文档已被其他进程修改，需要重试
}
```

#### 方案C：使用MongoDB事务（4.0+，需副本集）

对于更复杂的多文档操作，使用多文档事务：

```typescript
const session = await mongoose.startSession();
session.startTransaction();

try {
  const event = await Event.findOne({ id: eventID }).session(session);
  // 检查容量...
  // 修改attendee...
  await event.save({ session });
  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
} finally {
  session.endSession();
}
```

## 三、审核机制下的名额锁定合理性分析

### 3.1 当前容量计算逻辑

当前`getApprovedAttendeeCount`函数（[Event.ts:81-90](file:///e:/solo-code-2/gathio/src/models/Event.ts#L81-L90)）：

```typescript
export const getApprovedAttendeeCount = (
  event: Pick<IEvent, "attendees" | "approveRegistrations">
): number => {
  if (!event.attendees) return 0;
  return event.attendees.reduce((acc, a) => {
    if (a.status !== "attending") return acc;
    if (event.approveRegistrations && !a.approved) return acc;  // 只统计已批准
    return acc + (a.number || 1);
  }, 0);
};
```

**核心规则**：对于`approveRegistrations=true`的活动，只有`approved=true`的attendee才占用名额。

### 3.2 名额锁定的合理性讨论

#### 3.2.1 不锁定名额（当前实现）的问题

**场景**：活动上限100人，当前已批准80人，剩余20个名额。此时有50个用户提交申请，全部处于pending状态：

```
已批准: 80人
待审核: 50人 (approved=false)
剩余名额显示: 20人 (100 - 80)

当管理员批量审核时：
批准前20人 → 已批准=100，名额已满
拒绝后30人 → 用户体验极差（提交申请时显示有名额，审核时却被拒绝）
```

**问题**：
1. **用户体验差**：提交申请时显示有名额，但最终可能因名额不足被拒绝。
2. **管理难度大**：管理员审核时无法准确判断有多少名额可用。
3. **先到先得不公平**：审核顺序而非申请顺序决定名额归属。

#### 3.2.2 锁定名额（申请即占用）的合理性

**支持锁定的理由**：
1. **用户预期一致**：提交申请时看到的剩余名额就是真实可用的。
2. **先到先得公平**：申请时间顺序决定名额归属，而非审核顺序。
3. **避免超发风险**：不会出现申请数远超名额数的情况。

**反对锁定的理由**：
1. **名额浪费**：如果大量用户申请后被拒绝，名额会被"锁定"但最终未使用。
2. **恶意占用**：攻击者可批量提交无效申请占用名额。
3. **审核压力**：管理员需及时处理审核，否则名额被长期锁定。

#### 3.2.3 合理的折中方案

**方案：带超时的名额锁定**

```typescript
interface IAttendee {
  // ... 现有字段
  lockedUntil?: Date;  // 锁定过期时间
  lockExpired?: boolean;
}

// 容量计算逻辑修改为：
export const getEffectiveAttendeeCount = (event: IEvent): number => {
  if (!event.attendees) return 0;
  const now = new Date();
  return event.attendees.reduce((acc, a) => {
    if (a.status !== "attending") return acc;
    
    // 已批准的占用名额
    if (a.approved) return acc + (a.number || 1);
    
    // 待审核但在锁定时间内的，也占用名额
    if (event.approveRegistrations && !a.approved) {
      if (a.lockedUntil && a.lockedUntil > now && !a.lockExpired) {
        return acc + (a.number || 1);
      }
    }
    return acc;
  }, 0);
};
```

**机制说明**：
1. 用户提交申请时，名额被锁定24小时（可配置）。
2. 管理员需在锁定时间内完成审核。
3. 锁定超时后自动释放名额，其他用户可申请。
4. 需配合定时任务清理过期锁定。

**优点**：
- 兼顾用户体验和名额利用率
- 防止恶意长期占用
- 给管理员合理的审核窗口
