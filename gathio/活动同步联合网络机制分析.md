# 活动内容修订触发的联合网络同步机制分析

## 概述

本文分析 gathio 项目中 ActivityPub 联合网络同步机制的三个关键技术问题，基于 [event.ts](file:///e:/solo-code-2/gathio/src/routes/event.ts#L270-L582) 和 [activitypub.ts](file:///e:/solo-code-2/gathio/src/activitypub.ts) 的源代码。

## 问题一：大量关注者造成的Express路由阻塞延迟

### 问题分析

当活动拥有大量 Mastodon 关注者时，在 PUT 路由处理中串行调用广播函数会造成严重的请求阻塞。

### 核心代码位置

- [event.ts](file:///e:/solo-code-2/gathio/src/routes/event.ts#L483-L516) - PUT路由中的三次串行广播调用
- [activitypub.ts](file:///e:/solo-code-2/gathio/src/activitypub.ts#L281-L356) - `broadcastToFollowers` 函数的同步循环实现

### 技术细节

```typescript
// event.ts 第483-516行 - 三次串行广播调用
await broadcastCreateMessage(jsonObject, event.followers || [], eventID);
await broadcastUpdateMessage(jsonUpdateObject, event.followers || [], eventID);
await broadcastUpdateMessage(jsonEventObject, event.followers || [], eventID);

// activitypub.ts 第294-353行 - 同步for循环实现
for (const follower of followers) {
  // ... 数据处理
  try {
    await signAndSend(buildMessage(actorId, guid), eventID, targetDomain, inbox);
    // 每个 HTTP 请求都是阻塞的
  } catch (err) {
    // 单个错误仅记录日志
  }
}
```

### 造成的严重延迟问题

| 问题类型 | 具体表现 |
|---------|---------|
| **串行IO阻塞** | 每个 `signAndSend` 都包含一次 HTTP POST 请求，使用 `await` 在 `for...of` 循环中串行执行。若有 N 个关注者，总耗时 = 单次网络耗时 × N |
| **累积延迟放大** | 假设平均每个请求耗时 100ms，1000 个关注者就需要 100 秒才能完成所有广播，远超 HTTP 请求超时阈值 |
| **Express 工作线程占用** | Node.js 虽然是单线程，但每个请求上下文会占用事件循环资源，长时间阻塞会导致其他请求无法及时处理 |
| **用户体验极差** | 前端编辑操作会长时间挂起，用户可能会重复提交，造成重复编辑 |
| **网关/负载均衡器超时** | 前置的 Nginx 或 CDN 可能在 30-60 秒后断开连接，导致客户端收到 504 错误 |

### 根本原因

广播逻辑设计为**请求-响应式同步执行**，没有采用**异步队列+后台工作者**的架构模式。

## 问题二：网络抖动/超时对其他关注者的影响

### 问题分析

由于缺乏重试队列和容错机制，单个实例的网络问题是否会影响其他关注者。

### 核心代码位置

- [activitypub.ts](file:///e:/solo-code-2/gathio/src/activitypub.ts#L346-L353) - 单个 follower 的错误处理
- [event.ts](file:///e:/solo-code-2/gathio/src/routes/event.ts#L488-L503) - 广播函数的外层错误处理

### 技术细节

#### 1. 循环内的错误捕获（局部容错）

```typescript
// activitypub.ts 第346-353行
try {
  await signAndSend(buildMessage(actorId, guid), eventID, targetDomain, inbox);
  addToLog("broadcast", "success", `Sent to ${actorId}`);
  statuses.push(`sent to ${actorId}`);
} catch (err) {
  addToLog("broadcast", "error", `Didn't send to ${actorId} with error ${err}`);
  statuses.push(`Didn't send to ${actorId} with error ${err}`);
}
```

✅ **结论**：在循环内部，单个 follower 发送失败只会被记录日志，**不会中断整个循环**。因此，单个实例的网络问题**不会影响**同一个广播批次内的其他关注者。

#### 2. 外层错误处理（潜在风险）

```typescript
// event.ts 第483-490行
try {
  await broadcastCreateMessage(jsonObject, event.followers || [], eventID);
} catch (err) {
  return console.log("Error broadcasting create message:", err);
}
// 如果 broadcastCreateMessage 抛出异常，后续的 broadcastUpdateMessage 将不会执行！
```

⚠️ **风险点**：如果 `broadcastCreateMessage` 函数本身抛出异常（而非在循环内被捕获），会直接 `return`，导致后续的 `broadcastUpdateMessage` 不会执行，**所有关注者都收不到后续类型的更新消息**。

### 缺乏重试机制的后果

| 后果类型 | 具体影响 |
|---------|---------|
| **消息丢失** | 网络抖动导致失败的关注者不会收到任何通知，且没有重试机制保证最终一致性 |
| **状态不一致** | 部分关注者收到了更新，部分没收到，造成联邦网络中活动状态的分裂 |
| **静默失败** | 错误仅记录在本地日志中，管理员和活动创建者都无法感知 |
| **没有死信队列** | 失败的消息不会被持久化，无法事后重发 |

## 问题三：网络重发/乱序时的去重机制

### 问题分析

使用 `crypto.randomBytes` 生成的随机 ID 作为事件更新的指纹，远程实例如何处理重发和乱序。

### 核心代码位置

- [event.ts](file:///e:/solo-code-2/gathio/src/routes/event.ts#L472-L480) - Note ID 的生成
- [activitypub.ts](file:///e:/solo-code-2/gathio/src/activitypub.ts#L287) - 包装活动 ID 的生成
- [activitypub.ts](file:///e:/solo-code-2/gathio/src/activitypub.ts#L365-L373) - Create 活动 ID 生成

### 技术细节

```typescript
// event.ts 第472行 - 每次编辑生成全新的随机ID
const guidObject = crypto.randomBytes(16).toString("hex");
const jsonObject = {
  id: `https://${domain}/${eventID}/m/${guidObject}`,  // 每次都不同
  type: "Note",
  content: `${diffText} ...`,
};

// activitypub.ts 第365-373行 - 外层 Create 活动也使用随机ID
id: `https://${domain}/${eventID}/m/${guid}`,  // 每次广播都不同
```

### ActivityPub 标准的去重机制

根据 ActivityPub 规范，远程实例（如 Mastodon）的去重逻辑基于 **活动 ID (`id` 字段)**：

| 场景 | 远程实例行为 |
|-----|-------------|
| **完全相同的消息重发** | 收到 `id` 相同的活动时，Mastodon 等实现会直接忽略，认为是重复投递 |
| **网络重传导致重复包** | 如果是网络层重传，`id` 相同，会被正确去重 |
| **应用层重发（重试）** | 如果代码重试时重新生成了 `guid`，则 `id` 不同，远程实例会认为是新消息，**无法去重** |
| **乱序到达** | ActivityPub 不保证顺序，远程实例通常按接收时间处理。但由于每条更新的 `id` 都不同，即使内容相同也会被当作独立事件 |

### 关键问题：内容相同但 ID 不同的重复更新

当前实现中，**每次编辑操作都会生成全新的随机 ID**，这意味着：

1. ✅ **网络层重传**：TCP 重传的相同数据包会被正确去重（因为 `id` 相同）
2. ❌ **应用层重试**：如果添加重试逻辑但不固定 ID，重试发送的相同内容会被远程实例视为新的独立消息，导致重复显示
3. ❌ **快速连续编辑**：用户在短时间内多次编辑，每次都会生成独立的 Note，导致关注者时间线被刷屏
4. ❌ **基于内容的去重不存在**：远程实例不会基于 `content` 内容哈希去重，只信任 `id` 字段

### 改进方向

要实现可靠的去重，应该：

1. **基于内容哈希生成 ID**：使用 `content` 的 SHA-256 哈希作为 ID 的一部分，相同内容的更新始终具有相同的 ID
2. **添加时间戳版本号**：使用 `eventID + timestamp` 作为 ID，确保可排序且同一毫秒内的编辑不会冲突
3. **幂等性设计**：重试时保持相同的活动 ID，确保"至少一次"投递不会导致重复

---

## 总结

| 问题 | 现状评估 | 建议改进方向 |
|-----|---------|-------------|
| **路由阻塞延迟** | ❌ 严重问题，大量关注者时会导致请求超时 | 引入消息队列 + 后台 worker 异步处理广播 |
| **容错机制** | ⚠️ 部分容错，单个失败不影响同批次，但缺乏重试 | 实现持久化重试队列 + 死信队列 + 指数退避 |
| **去重机制** | ⚠️ 依赖随机 ID，仅能处理网络层重传 | 改用内容哈希或确定性 ID 生成策略 |
