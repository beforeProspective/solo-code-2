# 活动退订功能安全分析报告

## 概述

本文档对 gathio 项目中用户活动退订（unattend）功能进行安全分析，重点关注三个核心安全问题：明文密码传输、哈希碰撞风险、以及 CSRF 攻击风险。

---

## 1. 明文移除密码邮件传输安全性分析

### 问题描述

用户在 RSVP 时生成的 `removalPassword` 以明文形式通过邮件发送给用户，同时其 SHA256 哈希用于一键取消链接。这种设计是否符合安全防泄露原则？

### 代码证据

**邮件发送逻辑** - [routes/event.ts#L850-L863](file:///e:/solo-code-2/gathio/src/routes/event.ts#L850-L863)

```typescript
// Send confirmation email to attendee
if (attendeeEmail && attendee.approved) {
  try {
    await req.emailService.sendEmailFromTemplate({
      to: attendeeEmail,
      subject: i18next.t("routes.addeventattendeesubject", {
        eventName: event.name,
      }),
      templateName: "addEventAttendee",
      templateData: {
        eventID: req.params.eventID,
        removalPassword,           // 明文密码传递给模板
        removalPasswordHash: hashString(removalPassword),  // 哈希值
      },
    });
  } catch (e) {
    console.error("Error sending addEventAttendee email:", e);
  }
}
```

**邮件模板（明文密码）** - [addEventAttendeeText.handlebars#L5-L7](file:///e:/solo-code-2/gathio/views/emails/addEventAttendee/addEventAttendeeText.handlebars#L5-L7)

```handlebars
{{t "views.emails.addeventattendee.removelink" }}: https://{{domain}}/event/{{eventID}}/unattend/{{removalPasswordHash}}

{{t "views.emails.addeventattendee.removepassword" }}: {{removalPassword}}
```

**数据库存储（明文）** - [Event.ts#L105-L110](file:///e:/solo-code-2/gathio/src/models/Event.ts#L105-L110)

```typescript
removalPassword: {
  type: String,
  trim: true,
  unique: true,
  sparse: true,
},
```

### 安全分析

| 评估维度 | 风险等级 | 说明 |
|---------|---------|------|
| **邮件传输安全** | ⚠️ 中风险 | 邮件默认以明文传输，即使使用 TLS，邮件服务器仍可能缓存或记录邮件内容。若邮件服务器被攻破，所有用户的移除密码将泄露。 |
| **数据库存储安全** | ⚠️ 中高风险 | `removalPassword` 以明文形式存储在数据库中。数据库若发生数据泄露，攻击者可直接获取所有用户的移除密码。 |
| **浏览器历史记录** | ✅ 低风险 | 一键取消链接使用哈希值而非明文密码，避免了明文密码出现在浏览器历史记录中。 |

### 结论与建议

**不符合**最佳安全实践。虽然提供了一键取消链接（使用哈希），但明文密码的邮件传输和数据库存储存在重大安全隐患。

**改进建议：**
1. **停止在邮件中发送明文密码** - 仅发送包含哈希的一键取消链接即可
2. **数据库中仅存储哈希值** - 不应存储明文 `removalPassword`
3. **使用加盐哈希** - 对 `removalPassword` 使用随机盐值进行哈希，而非简单 SHA256

## 2. 哈希碰撞与哈希遍历攻击风险分析

### 问题描述

系统仅凭 `removalPasswordHash` 的哈希匹配就在 `Event.findOne` 中找到对应出席者并直接删除，是否存在由于哈希碰撞或遍历哈希导致其他用户被恶意退订的风险？

### 代码证据

**一键取消路由** - [routes/event.ts#L753-L786](file:///e:/solo-code-2/gathio/src/routes/event.ts#L753-L786)

```typescript
router.get(
  "/event/:eventID/unattend/:removalPasswordHash",
  async (req: Request, res: Response) => {
    // Find the attendee by the unattendPasswordHash
    const event = await Event.findOne({ id: req.params.eventID });
    if (!event) {
      return res.redirect("/404");
    }
    const attendee = event.attendees?.find(
      (o) =>
        hashString(o.removalPassword || "") === req.params.removalPasswordHash,
    );
    if (!attendee) {
      return res.redirect(`/${req.params.eventID}`);
    }
    // Remove the attendee from the event
    event.attendees = event.attendees?.filter(
      (o) => o.removalPassword !== attendee.removalPassword,
    );
    await event.save();
    // ... 发送确认邮件
    return res.redirect(`/${req.params.eventID}?m=unattend`);
  },
);
```

**哈希函数实现** - [generator.ts#L38-L39](file:///e:/solo-code-2/gathio/src/util/generator.ts#L38-L39)

```typescript
export const hashString = (input: string) =>
  crypto.createHash("sha256").update(input).digest("hex");
```

### 安全分析

#### 2.1 哈希碰撞攻击风险

| 攻击类型 | 可行性 | 说明 |
|---------|-------|------|
| **暴力碰撞** | ❌ 极低 | SHA256 具有 256 位输出空间，生日攻击需要约 2^128 次计算，在实际中不可行。 |
| **已知碰撞攻击** | ❌ 极低 | 目前 SHA256 尚未发现实际可行的碰撞攻击方法。 |
| **长度扩展攻击** | ❌ 不适用 | 此场景不涉及长度扩展攻击利用条件。 |

**结论**：SHA256 哈希碰撞风险 **极低**，在实际应用中可忽略。

---

#### 2.2 哈希遍历/暴力破解风险

| 攻击类型 | 可行性 | 说明 |
|---------|-------|------|
| **在线暴力破解** | ⚠️ 中风险 | 若 `removalPassword` 熵值不足，攻击者可对特定 `eventID` 进行暴力枚举。 |
| **离线破解（数据库泄露时）** | ⚠️ 高风险 | 由于使用简单 SHA256（无盐），数据库泄露后攻击者可使用彩虹表快速破解所有哈希值。 |

**关键问题**：

1. **缺少速率限制** - 该 GET 端点没有任何速率限制或验证码保护，攻击者可以快速枚举哈希值
2. **无盐哈希** - `hashString()` 使用简单 SHA256 加盐，相同输入产生相同输出，易受彩虹表攻击

### 结论与建议

**哈希碰撞风险极低**，但**哈希遍历/暴力破解风险存在**，特别是在数据库泄露场景下。

**改进建议：**
1. **添加速率限制** - 对 `/event/:eventID/unattend/:removalPasswordHash` 端点实施速率限制
2. **使用加盐哈希** - 使用 `crypto.scrypt` 或 `bcrypt` 等带盐的密码哈希算法
3. **增加移除密码熵值** - 确保 `removalPassword` 具有足够的随机性和长度（当前生成逻辑需验证）

## 3. CSRF 与点击劫持攻击风险分析

### 问题描述

取消操作（DELETE 或 GET 一键链接）不需要额外的图形验证码或 CsrfToken 拦截，攻击者是否可以通过欺骗用户点击类似链接来实施大面积恶意 RSVP 清除？

### 代码证据

**DELETE 端点（无 CSRF 保护）** - [routes/event.ts#L686-L750](file:///e:/solo-code-2/gathio/src/routes/event.ts#L686-L750)

```typescript
router.delete(
  "/event/attendee/:eventID",
  async (req: Request, res: Response) => {
    const removalPassword = req.query.p;
    if (!removalPassword) {
      return res
        .status(400)
        .json({ error: "Please provide a removal password." });
    }
    // 直接根据 removalPassword 删除，无 CSRF 检查
    const response = await Event.findOne({
      id: req.params.eventID,
      "attendees.removalPassword": removalPassword,
    });
    // ... 删除逻辑
  },
);
```

**GET 一键取消端点（无保护）** - [routes/event.ts#L753-L786](file:///e:/solo-code-2/gathio/src/routes/event.ts#L753-L786)

```typescript
router.get(
  "/event/:eventID/unattend/:removalPasswordHash",
  async (req: Request, res: Response) => {
    // 无 CSRF token 验证
    // 无验证码
    // 无二次确认页面
    const event = await Event.findOne({ id: req.params.eventID });
    // ... 直接删除逻辑
    await event.save();
    return res.redirect(`/${req.params.eventID}?m=unattend`);
  },
);
```

### 安全分析

#### 3.1 GET 请求攻击风险

| 攻击向量 | 风险等级 | 说明 |
|---------|---------|------|
| **CSRF 攻击** | ⚠️ 高风险 | GET 请求天然易受 CSRF 攻击。攻击者可构造 `<img>`、`<iframe>` 标签或直接发送链接诱导用户点击，触发退订操作。 |
| **钓鱼攻击** | ⚠️ 高风险 | 攻击者可通过邮件、社交工程等方式诱导用户点击退订链接，实施"退订轰炸"。 |
| **点击劫持** | ⚠️ 中风险 | 若网站未设置 `X-Frame-Options` 头，攻击者可通过 iframe 隐藏的方式诱导用户点击。 |

**攻击场景示例：**

```html
<!-- 攻击者网站中的恶意代码 -->
<img src="https://target.com/event/ABC123/unattend/哈希值" style="display:none" />

<!-- 或通过社交工程 -->
<a href="https://target.com/event/ABC123/unattend/哈希值">
  点击查看活动详情
</a>
```

用户访问攻击者页面或点击链接时，会**自动且静默**地从活动中退订。

---

#### 3.2 DELETE 请求攻击风险

| 攻击向量 | 风险等级 | 说明 |
|---------|---------|------|
| **CSRF 攻击** | ⚠️ 中风险 | 虽然是 DELETE 请求，但攻击者仍可通过构造恶意表单或使用 `fetch` API 进行 CSRF 攻击。 |
| **跨域请求** | ✅ 低风险 | 浏览器同源策略通常会阻止跨域 DELETE 请求，除非服务器配置了宽松的 CORS。 |

### 结论与建议

**存在高风险**。GET 一键取消链接缺乏必要的保护机制，容易被滥用于恶意退订。

**改进建议：**
1. **添加二次确认页面** - GET 请求不应直接执行删除，应先显示确认页面，用户点击确认后再执行 POST/DELETE
2. **实施 CSRF 保护** - 对 DELETE/POST 端点添加 CSRF token 验证
3. **添加速率限制** - 防止批量枚举攻击
4. **考虑添加验证码** - 对于高风险操作或检测到异常行为时要求验证码
5. **设置安全头** - 添加 `X-Frame-Options: DENY` 防止点击劫持

## 总体安全评估

### 风险矩阵

| 安全问题 | 风险等级 | 影响范围 | 修复难度 |
|---------|---------|---------|---------|
| 明文密码邮件传输 | ⚠️ 中 | 所有用户 | 低 |
| 明文密码数据库存储 | ⚠️ 中高 | 所有用户 | 中 |
| 无盐 SHA256 哈希 | ⚠️ 中 | 数据库泄露时 | 中 |
| GET 直接执行删除 | ⚠️ 高 | 所有用户 | 低 |
| 缺少 CSRF 保护 | ⚠️ 高 | 所有用户 | 低 |
| 缺少速率限制 | ⚠️ 中 | 所有用户 | 低 |

### 优先级建议

**高优先级修复：**
1. 将 GET 一键取消改为先显示确认页面，POST 提交确认
2. 停止在邮件中发送明文 `removalPassword`
3. 对退订端点添加速率限制

**中优先级修复：**
1. 数据库中仅存储哈希值，不存储明文 `removalPassword`
2. 使用带盐的密码哈希算法（如 bcrypt、scrypt）
3. 添加 CSRF token 验证

**低优先级增强：**
1. 异常行为检测时要求验证码
2. 点击劫持保护头
