# 私有站点模式下用户信息嗅探漏洞安全分析报告

## 漏洞概述

在私有站点配置（`settings.PUBLIC = False`）下，由于中间件路径过滤不严格和ViewSet权限控制缺失，未认证用户可以通过特定API接口嗅探系统中已注册用户的敏感信息。

---

## 问题1：未认证用户能否嗅探用户信息？

### 结论：**可以，且存在严重的信息泄露风险**

### 技术分析

#### 1.1 Public中间件的路径过滤机制

[middleware.py](file:///e:/solo-code-2/pinry/users/middleware.py#L6-L16) 中的 `Public` 中间件定义：

```python
class Public(MiddlewareMixin):
    acceptable_paths = (
        "/api/v2/profile/",
    )

    def process_request(self, request):
        if settings.PUBLIC is False and not request.user.is_authenticated:
            for path in self.acceptable_paths:
                if not request.path.startswith(path):
                    return HttpResponseForbidden()
```

**关键问题**：
- 白名单仅检查路径前缀，所有以 `/api/v2/profile/` 开头的路径都被允许
- 根据 [urls.py](file:///e:/solo-code-2/pinry/pinry/urls.py#L22)，该路径下包含了整个 `users.urls`
- 在 [views.py](file:///e:/solo-code-2/pinry/users/views.py#L104-L106) 中注册了两个路由：
  - `/api/v2/profile/users/` → `UserViewSet`
  - `/api/v2/profile/public-users/` → `PublicUserViewSet`

#### 1.2 公共用户查询接口的可用性

`PublicUserViewSet` 位于 [views.py](file:///e:/solo-code-2/pinry/users/views.py#L24-L36)：

```python
class PublicUserViewSet(
    mixins.RetrieveModelMixin,
    mixins.ListModelMixin,
    GenericViewSet,
):
    serializer_class = UserSerializer
    filter_backends = (DjangoFilterBackend, )
    filter_fields = ("username", )
    pagination_class = None

    def get_queryset(self):
        username = self.request.GET.get("username", "")
        return User.objects.filter(username=username)
```

**攻击路径**：

| 攻击方式 | HTTP请求 | 效果 |
|---------|---------|------|
| 精确查询 | `GET /api/v2/profile/public-users/?username=admin` | 返回匹配用户的完整信息 |
| 模糊匹配 | `GET /api/v2/profile/public-users/?username__contains=adm` | 可用于用户名枚举 |
| 列表遍历 | `GET /api/v2/profile/public-users/` | 由于`get_queryset`需要username参数，返回空列表 |

## 问题2：字段序列化与信息泄露分析

### 2.1 UserSerializer序列化的字段

[serializers.py](file:///e:/solo-code-2/pinry/users/serializers.py#L24-L40) 中的 `UserSerializer`：

```python
class UserSerializer(serializers.HyperlinkedModelSerializer):
    class Meta:
        model = User
        fields = (
            'username',      # 用户名
            'token',         # API访问令牌（条件返回）
            'email',         # 邮箱地址 ⚠️
            'gravatar',      # Gravatar头像哈希 ⚠️
            'password',      # 密码（只写）
            'password_repeat',  # 密码确认（只写）
            settings.DRF_URL_FIELD_NAME,
        )
```

### 2.2 敏感字段泄露分析

| 字段 | 泄露风险 | 泄露条件 | 说明 |
|-----|---------|---------|------|
| **username** | 高 | 任何情况 | 用户名直接暴露 |
| **email** | 高 | 任何情况 | **未做任何权限控制，直接返回** |
| **gravatar** | 高 | 任何情况 | 由email哈希计算得出，泄露email等价于泄露gravatar |
| **token** | 中 | 仅自身用户 | `get_token`方法检查了`request.user == obj` |

### 2.3 Gravatar哈希的安全隐患

在 [models.py](file:///e:/solo-code-2/pinry/users/models.py#L19-L21) 中：

```python
@property
def gravatar(self):
    return hashlib.md5(self.email.encode('utf-8')).hexdigest()
```

**安全风险**：
- Gravatar哈希是MD5(email.lower().strip())
- 虽然是单向哈希，但由于邮箱格式有限，可通过彩虹表反向查询
- 已知用户名的情况下，可暴力枚举常见邮箱后缀
- 直接泄露email使gravatar保护失去意义

## 问题3：安全加固方案

### 方案一：改造Public中间件（推荐）

#### 修改 [middleware.py](file:///e:/solo-code-2/pinry/users/middleware.py)

```python
from django.conf import settings
from django.http import HttpResponseForbidden
from django.utils.deprecation import MiddlewareMixin


class Public(MiddlewareMixin):
    acceptable_paths = (
        "/api/v2/profile/login/",
        "/api/v2/profile/users/",
    )

    def process_request(self, request):
        if settings.PUBLIC is False and not request.user.is_authenticated:
            path_allowed = False
            for path in self.acceptable_paths:
                if request.path.startswith(path):
                    path_allowed = True
                    break
            if not path_allowed:
                return HttpResponseForbidden()
```

**改进要点**：
- 将白名单从 `/api/v2/profile/` 细化到具体路径
- 仅开放登录接口和用户注册/自我管理接口
- 明确排除 `/api/v2/profile/public-users/` 路径

---

### 方案二：改造PublicUserViewSet（必要）

#### 修改 [views.py](file:///e:/solo-code-2/pinry/users/views.py#L24-L36)

```python
class PublicUserViewSet(
    mixins.RetrieveModelMixin,
    mixins.ListModelMixin,
    GenericViewSet,
):
    serializer_class = PublicUserSerializer  # 使用Public而非User序列化器
    filter_backends = (DjangoFilterBackend, )
    filter_fields = ("username", )
    pagination_class = None

    def get_queryset(self):
        # 私有模式下，未认证用户禁止查询
        if not settings.PUBLIC and not self.request.user.is_authenticated:
            return User.objects.none()
        
        username = self.request.GET.get("username", "")
        if username:
            return User.objects.filter(username=username)
        return User.objects.all()
```

**改进要点**：
- 私有模式下未认证用户返回空查询集
- 使用`PublicUserSerializer`替代`UserSerializer`

### 方案三：修复序列化器字段权限（必须）

### 方案四：综合加固方案（最佳实践）

结合以上三种方案，实现**纵深防御**：

1. **网络层**：Public中间件精确控制可访问路径
2. **权限层**：ViewSet中检查settings.PUBLIC和用户认证状态
3. **数据层**：使用正确的序列化器，仅暴露必要字段

#### 完整修复后的PublicUserViewSet：

```python
from rest_framework.permissions import BasePermission, IsAuthenticatedOrReadOnly

class PublicUserPermission(BasePermission):
    def has_permission(self, request, view):
        if settings.PUBLIC:
            return True
        return request.user.is_authenticated

class PublicUserViewSet(
    mixins.RetrieveModelMixin,
    mixins.ListModelMixin,
    GenericViewSet,
):
    permission_classes = [PublicUserPermission]
    serializer_class = PublicUserSerializer
    filter_backends = (DjangoFilterBackend, )
    filter_fields = ("username", )
    pagination_class = None

    def get_queryset(self):
        username = self.request.GET.get("username", "")
        if username:
            return User.objects.filter(username=username)
        return User.objects.all()
```

## 漏洞严重程度评估

| 评估维度 | 等级 | 说明 |
|---------|------|------|
| 攻击难度 | 极低 | 无需认证，构造简单HTTP请求即可 |
| 影响范围 | 高 | 所有已注册用户的邮箱和用户名 |
| 利用后果 | 中 | 可用于钓鱼攻击、账号枚举、社会工程 |
| 修复成本 | 低 | 代码改动量小，无业务影响 |

**综合评级：高危**
