# Pinry SPA 登录态同步与响应拦截器分析

## 问题分析

### 1. 多标签页localStorage缓存不同步问题

#### 问题描述
当用户在浏览器的一个标签页中退出登录后，其他已打开的标签页中的`localStorage`缓存未能同步失效，导致UI状态与后端真实登录状态不一致。

#### 代码依据
在 [api.js](file:///e:/solo-code-2/pinry/pinry-spa/src/components/api.js#L245-L272) 的 `User.fetchUserInfo` 方法中：

```javascript
fetchUserInfo(force = false) {
  const self = this;
  if (!force) {
    const userInfo = storage.get(self.storageKey);  // 优先读取本地缓存
    if (userInfo !== null) {
      return new Promise(resolve => resolve(userInfo));
    }
  }
  // ... 发起网络请求获取用户信息
}
```

在 [api.js](file:///e:/solo-code-2/pinry/pinry-spa/src/components/api.js#L215-L227) 的 `User.logOut` 方法中：

```javascript
logOut() {
  const self = this;
  return new Promise((resolve) => {
    axios.get('/api-auth/logout/').then(() => {
      storage.set(self.storageKey, null, 1);  // 仅清除当前标签页缓存
      resolve();
    });
  });
}
```

#### 导致的UI状态不一致问题

| 场景 | 表现 | 用户体验问题 |
|------|------|-------------|
| **头部导航栏** | 仍然显示"已登录"状态，显示用户名和"退出"按钮，而实际上用户已在其他标签页登出 | 用户误以为自己仍处于登录状态 |
| **权限按钮显示** | "创建Pin"、"管理Board"等需要登录的按钮仍然可见且可点击 | 用户点击后操作失败，产生困惑 |
| **用户信息展示** | 用户头像、用户名等个人信息仍然展示在页面上 | 与实际状态不符，产生数据不一致的错觉 |
| **缓存数据复用** | 调用 `fetchUserInfo()` 时直接返回缓存的用户数据，不会发起真实请求验证 | 所有依赖用户状态的组件都处于错误状态 |

### 2. 未配置Axios响应拦截器导致的用户体验问题

#### 问题描述
[csrf.js](file:///e:/solo-code-2/pinry/pinry-spa/src/components/utils/csrf.js#L22-L35) 仅配置了请求拦截器用于CSRF令牌注入，但没有配置全局响应拦截器来统一处理 `401 Unauthorized` 和 `403 Forbidden` 状态码。

#### 当前CSRF配置
```javascript
function setUpAxiosCsrfConfig() {
  axios.interceptors.request.use(
    (config) => {
      if (!csrfSafeMethod(config.method.toUpperCase())) {
        config.headers['X-CSRFToken'] = getCSRFToken();
      }
      return config;
    },
    (error) => {
      Promise.reject(error);  // 错误处理有bug！缺少return
    }
  );
}
```

#### 当后端会话过期时前端的反应

以创建Pin图片请求为例 [api.js](file:///e:/solo-code-2/pinry/pinry-spa/src/components/api.js#L71-L77)：

```javascript
Pin.create(jsonData) {
  const url = `${API_PREFIX}pins/`;
  return axios.post(url, jsonData);  // 没有统一的错误处理
}
```

##### 用户体验弊端：

1. **静默失败**
   - 请求返回401/403后，由于没有统一拦截处理，错误被传递到各个业务组件
   - 如果业务组件遗漏了错误处理，页面可能没有任何视觉反馈
   - 用户不知道操作失败的原因

2. **错误提示不一致**
   - 不同组件可能有各自的错误处理方式，提示信息风格不统一
   - 有的可能只在控制台打印错误，用户不可见

3. **无法自动引导**
   - 无法在检测到未授权时自动跳转到登录页面
   - 用户需要手动刷新页面或重新导航才能发现自己已登出

4. **操作状态残留**
   - 表单可能停留在"提交中"状态，加载动画无限旋转
   - 用户无法进行下一步操作，也不知道问题出在哪里

5. **级联错误**
   - 后续依赖该操作结果的请求可能连续失败
   - 产生多个错误日志，增加排查难度

---

### 3. 解决方案：Axios响应拦截器 + Storage事件监听

#### 方案一：配置Axios全局响应拦截器

在 [csrf.js](file:///e:/solo-code-2/pinry/pinry-spa/src/components/utils/csrf.js) 中添加响应拦截器：

```javascript
import axios from 'axios';
import router from '../../router';
import storage from './storage';

function setUpAxiosCsrfConfig() {
  // 请求拦截器（原有CSRF逻辑）
  axios.interceptors.request.use(
    (config) => {
      if (!csrfSafeMethod(config.method.toUpperCase())) {
        config.headers['X-CSRFToken'] = getCSRFToken();
      }
      return config;
    },
    (error) => {
      return Promise.reject(error);  // 修复：添加return
    }
  );

  // ===== 新增：响应拦截器 =====
  axios.interceptors.response.use(
    (response) => {
      return response;
    },
    (error) => {
      const { response } = error;
      
      if (response) {
        switch (response.status) {
          case 401:
          case 403:
            handleUnauthorized();
            break;
        }
      }
      
      return Promise.reject(error);
    }
  );
}

function handleUnauthorized() {
  const User = { storageKey: 'pinry.user' };
  
  storage.set(User.storageKey, null, 1);
  
  const currentPath = window.location.pathname;
  if (currentPath !== '/') {
    router.push('/');
  }
}
```

#### 方案二：监听Storage事件实现多标签页同步

在应用入口（如 `main.js` 或 `App.vue`）中添加Storage事件监听：

```javascript
import storage from './components/utils/storage';

const USER_STORAGE_KEY = 'pinry.user';

function setUpStorageSync() {
  window.addEventListener('storage', (event) => {
    if (event.key === USER_STORAGE_KEY) {
      handleUserStorageChange(event.newValue, event.oldValue);
    }
  });
}

function handleUserStorageChange(newValueStr, oldValueStr) {
  try {
    const newValue = newValueStr ? JSON.parse(newValueStr) : null;
    const oldValue = oldValueStr ? JSON.parse(oldValueStr) : null;
    
    if (oldValue && oldValue.value && (!newValue || !newValue.value)) {
      window.location.reload();
    }
  } catch (e) {
    console.error('Failed to parse storage change:', e);
  }
}
```

#### 方案三：增强的完整实现

**步骤1：创建统一的认证管理模块**

```javascript
// pinry-spa/src/components/utils/auth.js
import storage from './storage';
import router from '../../router';

const USER_STORAGE_KEY = 'pinry.user';

const Auth = {
  onUserLoggedOut: null,

  clearUserCache() {
    storage.set(USER_STORAGE_KEY, null, 1);
  },

  getUserFromCache() {
    return storage.get(USER_STORAGE_KEY);
  },

  isUserLoggedIn() {
    return this.getUserFromCache() !== null;
  },

  handleLogout() {
    this.clearUserCache();
    
    if (this.onUserLoggedOut) {
      this.onUserLoggedOut();
    }
    
    const currentPath = window.location.pathname;
    if (!['/', '/login'].includes(currentPath)) {
      router.push('/');
    }
  },

  setUpStorageListener() {
    window.addEventListener('storage', (event) => {
      if (event.key === USER_STORAGE_KEY) {
        const oldData = event.oldValue ? JSON.parse(event.oldValue) : null;
        const newData = event.newValue ? JSON.parse(event.newValue) : null;
        
        const wasLoggedIn = oldData && oldData.value;
        const isLoggedIn = newData && newData.value;
        
        if (wasLoggedIn && !isLoggedIn) {
          window.location.reload();
        }
      }
    });
  }
};

export default Auth;
```

**步骤2：在csrf.js中集成响应拦截器**

```javascript
import axios from 'axios';
import Auth from './auth';

function setUpAxiosCsrfConfig() {
  axios.interceptors.request.use(
    (config) => {
      if (!csrfSafeMethod(config.method.toUpperCase())) {
        config.headers['X-CSRFToken'] = getCSRFToken();
      }
      return config;
    },
    (error) => Promise.reject(error)
  );

  axios.interceptors.response.use(
    (response) => response,
    (error) => {
      if (error.response) {
        const { status } = error.response;
        
        if (status === 401 || status === 403) {
          Auth.handleLogout();
        }
      }
      return Promise.reject(error);
    }
  );
}
```

**步骤3：在应用入口初始化**

```javascript
// main.js 或 App.vue
import Auth from './components/utils/auth';

Auth.setUpStorageListener();
```

#### 方案效果总结

| 方案 | 解决的问题 | 优势 |
|------|-----------|------|
| **响应拦截器** | 统一处理401/403错误 | 自动清除缓存、自动跳转、用户体验一致 |
| **Storage事件监听** | 多标签页状态同步 | 无需轮询、实时同步、性能开销小 |
| **统一Auth模块** | 代码可维护性 | 逻辑集中、易于扩展、便于测试 |

#### 注意事项

1. **Storage事件限制**：Storage事件仅在**其他标签页**修改storage时触发，当前标签页修改不会触发
2. **路由守卫**：可配合Vue路由守卫进行前置登录检查
3. **令牌刷新**：如需支持refresh token，可在401时先尝试刷新令牌，失败再登出
4. **用户提示**：可在检测到登出时显示友好提示，如"您已在其他地方登出"
