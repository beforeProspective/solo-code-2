## 问题2：人工卡片任务被管理员点击完成（mark_completed）时，系统如何通知被挂起的User重新触发后续关联条件？

### 分析结论

系统通过**条件触发机制**实现后续关联条件的重新触发，核心流程是：检查条件依赖 → 验证全部完成 → 调用process_condition触发后续流程。

### 详细流程

#### 1. AdminTask.mark_completed() 方法

在 [admin_tasks/models.py#L205-L242](file:///e:/solo-code-2/ChiefOnboarding/back/admin/admin_tasks/models.py#L205-L242)：

```python
def mark_completed(self):
    from admin.sequences.tasks import process_condition

    self.completed = True
    self.save()

    # 步骤1：注册手动集成（如果是集成相关任务）
    if self.manual_integration is not None:
        self.manual_integration.register_manual_integration_run(self.new_hire)

    # 步骤2：查找以该AdminTask为触发条件的所有Condition
    conditions = self.new_hire.conditions.filter(
        condition_admin_tasks=self.based_on
    )

    # 步骤3：对每个依赖该任务的Condition进行检查
    for condition in conditions:
        condition_admin_tasks_id = condition.condition_admin_tasks.values_list(
            "id", flat=True
        )

        # 步骤4：验证该Condition下的所有AdminTask是否都已完成
        completed_tasks = AdminTask.objects.filter(
            based_on_id__in=condition_admin_tasks_id,
            new_hire=self.new_hire,
            completed=True,
        )

        # 步骤5：如果全部完成，则触发后续流程
        if completed_tasks.count() == len(condition_admin_tasks_id):
            process_condition(
                condition.id, self.new_hire.id, self.new_hire.has_slack_account
            )
```

#### 2. process_condition 任务函数

在 [sequences/tasks.py#L20-L130](file:///e:/solo-code-2/ChiefOnboarding/back/admin/sequences/tasks.py#L20-L130)：

在 [sequences/models.py#L881-L912](file:///e:/solo-code-2/ChiefOnboarding/back/admin/sequences/models.py#L881-L912)：

```python
def process_condition(self, user, skip_notification=False):
    # 执行所有直接关联的项目
    for field in ["to_do", "resources", "badges", "appointments", "introductions", "preboarding"]:
        for item in getattr(self, field).all():
            getattr(user, field).add(item)
            # 创建通知

    # 执行需要特殊处理的项目
    for field in ["admin_tasks", "external_messages", "integration_configs", "hardware"]:
        for item in getattr(self, field).all():
            item.execute(user)
```

### 触发链路总结

```
AdminTask.mark_completed()
    ↓
注册集成/硬件处理
    ↓
查找依赖此Task的Conditions
    ↓
验证Condition下所有Tasks是否全部完成？
    ├─ 否 → 不触发，等待其他任务完成
    └─ 是 → 调用 process_condition()
                        ↓
                Condition.process_condition()
                        ↓

                        ↓
                发送通知给User
                        ↓
                更新User进度
```

## 问题3：在人工任务的生命周期中，如果指派任务的管理员被中途删除，系统如何防止该挂起的集成任务变为没有任何归属人的死锁任务？

### 分析结论

**当前系统存在死锁风险，缺乏有效的防止机制。**

### 代码分析

#### 1. AdminTask.assigned_to 字段配置

在 [admin_tasks/models.py#L80-L87](file:///e:/solo-code-2/ChiefOnboarding/back/admin/admin_tasks/models.py#L80-L87)：

```python
assigned_to = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    verbose_name=_("Assigned to"),
    on_delete=models.CASCADE,  # 级联删除
    related_name="owner",
    blank=True,
    null=True,
)
```

**关键问题**：`on_delete=models.CASCADE` 配置存在，但字段又允许 `null=True`，这意味着：

- 当管理员被删除时，由于 `null=True` 会让Django将 `assigned_to` 设置为 `NULL` 而不是删除任务本身。

#### 2. 缺乏的保护机制

**缺失的机制包括：

1. **缺少重新分配机制**：系统没有信号（signal）监听用户删除事件，无法自动将任务重新分配给其他管理员

2. **缺少任务认领机制**：没有提供界面让其他管理员可以认领无负责人的任务

3. **缺少定时巡检机制**：没有定时任务检查 `assigned_to=None 的任务

4. **缺少兜底负责人**：没有设置默认负责人或组织负责人机制

#### 4. 潜在死锁场景

```
管理员被删除
    ↓
AdminTask.assigned_to → NULL
    ↓
任务仍处于未完成状态
    ↓
无人收到通知
```
