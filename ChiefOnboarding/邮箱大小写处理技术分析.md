## 一、PostgreSQL 中 iexact 查询的索引问题与优化方案

### 1.1 iexact 查询的工作原理

在 [CustomUserManager.get_by_natural_key](file:///e:/solo-code-2/ChiefOnboarding/back/users/models.py#L46-L48) 方法中：

```python
def get_by_natural_key(self, email):
    return self.get(**{self.model.USERNAME_FIELD + "__iexact": email})
```

Django 的 `iexact` 查找在 PostgreSQL 中会被转换为类似以下的 SQL：

```sql
SELECT * FROM users_user WHERE UPPER(email) = UPPER('user@example.com');
```

或者使用 `ILIKE` 操作符：

```sql
SELECT * FROM users_user WHERE email ILIKE 'user@example.com';
```

### 1.2 标准 B-tree 索引失效问题

**结论：是的，标准的 B-tree 索引会失效。**

#### 原因分析：

1. **函数调用破坏索引匹配**：当对列应用函数（如 `UPPER()`、`LOWER()`）时，PostgreSQL 查询规划器无法使用基于原始列值构建的标准 B-tree 索引。

2. **执行计划变化**：
   - 有索引但无法使用 → 全表扫描（Sequential Scan）
   - 性能影响：O(n) 时间复杂度，用户量大时查询变慢

#### 验证方法：

```sql
-- 创建测试表和索引
CREATE TABLE test_email (id SERIAL PRIMARY KEY, email VARCHAR(255) UNIQUE);
CREATE INDEX idx_test_email ON test_email(email);

-- 分析执行计划
EXPLAIN ANALYZE SELECT * FROM test_email WHERE email ILIKE 'test@example.com';
-- 结果: Seq Scan on test_email
```

---

### 1.3 优化方案对比

| 方案 | 实现方式 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **表达式索引** | `CREATE INDEX ON users_user(LOWER(email));` | 查询性能最优，无需改业务代码 | 索引体积稍大 | 大多数场景 |
| **citext 数据类型** | 改用 `citext` 类型列 | 最简洁，天然大小写不敏感 | 需要扩展，迁移成本高 | 新项目或大规模重构 |
| **排序规则（Collation）** | `COLLATE "en_US.utf8"` | 标准 SQL 方式 | 配置复杂，跨库兼容性差 | 特定语言环境 |

---

### 1.4 推荐方案：表达式索引

#### Django 迁移实现：

```python
# users/migrations/XXXX_add_lower_email_index.py
from django.db import migrations

class Migration(migrations.Migration):
    dependencies = [
        ('users', '00XX_previous_migration'),
    ]

    operations = [
        migrations.RunSQL(
            sql="CREATE INDEX users_user_email_lower_idx ON users_user (LOWER(email));",
            reverse_sql="DROP INDEX users_user_email_lower_idx;",
        ),
    ]
```

#### 或者使用 Django 3.2+ 的 `Index` 表达式：

```python
# models.py
from django.db.models import Index
from django.db.models.functions import Lower

class User(AbstractBaseUser):
    # ... 字段定义 ...

    class Meta:
        indexes = [
            Index(Lower('email'), name='users_user_email_lower_idx'),
        ]
```

#### 效果验证：

```sql
EXPLAIN ANALYZE SELECT * FROM users_user WHERE LOWER(email) = 'user@example.com';
-- 结果: Index Scan using users_user_email_lower_idx on users_user
```

## 二、并发注册时的脏数据问题与防范

### 2.1 当前实现的问题

在 [User.save()](file:///e:/solo-code-2/ChiefOnboarding/back/users/models.py#L413-L421) 方法中：

```python
def save(self, *args, **kwargs):
    self.email = self.email.lower()  # 仅在 Python 层转换
    # ...
    super(User, self).save(*args, **kwargs)
```

#### 竞态条件时序：

| 时间 | 线程 A | 线程 B | 数据库状态 |
|------|--------|--------|-----------|
| T1 | 接收 `User@Example.COM` | 接收 `user@example.com` | 无记录 |
| T2 | `email.lower()` → `user@example.com` | `email.lower()` → `user@example.com` | 无记录 |
| T3 | 检查唯一性 → 不存在 | 检查唯一性 → 不存在 | 无记录 |
| T4 | 执行 INSERT | 等待 | 插入 user@example.com |
| T5 | 提交成功 | 执行 INSERT | user@example.com |
| T6 | | 唯一性冲突？ | **取决于约束类型** |

---

### 2.2 可能产生的脏数据类型

#### 类型 1：重复邮箱记录（大小写变体）

**发生条件**：数据库层面只有 `UNIQUE(email)` 约束，但使用了大小写敏感的排序规则

**结果**：
- `user@example.com` 和 `User@Example.COM` 同时存在
- 登录时 `iexact` 会抛出 `MultipleObjectsReturned` 异常
- 用户数据分裂，权限、资源归属混乱

#### 类型 2：幽灵账户

**发生条件**：并发创建时事务隔离级别较低

**结果**：
- 一个邮箱对应多个用户 ID
- 审计日志、操作历史混乱

#### 类型 3：认证系统不可用

**发生条件**：`get_by_natural_key` 因重复记录抛出异常

**结果**：
- 所有使用该邮箱登录的用户都无法认证
- 影响范围：该邮箱的所有变体

---

### 2.3 防范措施

#### 措施 1：数据库层面的大小写无关唯一约束

```sql
-- PostgreSQL: 使用表达式唯一索引
CREATE UNIQUE INDEX users_user_email_lower_unq_idx 
ON users_user (LOWER(email));
```

Django 迁移实现：

```python
migrations.RunSQL(
    sql="""
        CREATE UNIQUE INDEX users_user_email_lower_unq_idx 
        ON users_user (LOWER(email));
    """,
    reverse_sql="DROP INDEX users_user_email_lower_unq_idx;",
)
```

#### 措施 2：使用 citext 数据类型（PostgreSQL 专用）

```sql
-- 首先启用扩展
CREATE EXTENSION IF NOT EXISTS citext;

-- 修改列类型
ALTER TABLE users_user ALTER COLUMN email TYPE citext;
```

#### 措施 3：应用层加锁（可选补充）

```python
from django.db import transaction, IntegrityError

@transaction.atomic
def create_user(email, **kwargs):
    email = email.lower()
    try:
        with transaction.atomic():
            return User.objects.create(email=email, **kwargs)
    except IntegrityError as e:
        if 'unique constraint' in str(e).lower():
            raise ValueError("邮箱已存在")
        raise
```

#### 措施 4：API 输入规范化（第一道防线）

在序列化器和视图中尽早规范化邮箱：

```python
# serializers.py
class UserSerializer(serializers.ModelSerializer):
    def validate_email(self, value):
        return value.lower().strip()
```

## 三、对 Django 认证和序列化系统的侵入性影响

### 3.1 Django 自然键机制回顾

Django 的自然键（Natural Key）用于：
- 序列化/反序列化（`dumpdata`/`loaddata`）
- 认证后端的用户查找
- 数据导入导出
- 外键引用的人类可读表示

### 3.2 对认证系统的影响

Django 的 `ModelBackend.authenticate()` 调用链：

### 3.4 其他侵入性影响

#### 影响点 1：DRF 序列化器

```python
class UserSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ['email', ...]
        extra_kwargs = {
            'email': {
                'validators': [UniqueValidator(queryset=User.objects.all())]
            }
        }
```

**问题**：DRF 的 `UniqueValidator` 是大小写敏感的，需要自定义：

```python
from rest_framework.validators import UniqueValidator

class CaseInsensitiveUniqueValidator(UniqueValidator):
    def filter_queryset(self, value, queryset):
        return queryset.filter(email__iexact=value)
```

#### 影响点 2：Admin 搜索

```python
# admin.py
class UserAdmin(admin.ModelAdmin):
    search_fields = ['email']  # 默认大小写敏感
```

需要改为：

```python
from django.db.models.functions import Lower

class UserAdmin(admin.ModelAdmin):
    def get_search_results(self, request, queryset, search_term):
        queryset, use_distinct = super().get_search_results(
            request, queryset, search_term
        )
        if search_term:
            queryset |= self.model.objects.filter(email__iexact=search_term)
        return queryset, use_distinct
```

## 四、综合改进建议

### 4.1 数据库层改进

1. **添加表达式唯一索引**（最高优先级）

### 5.2 改进的 UserManager

```python
class CustomUserManager(BaseUserManager):
    def get_by_natural_key(self, email):
        """
        大小写不敏感的用户查找。
        
        注意：数据库层面必须有 LOWER(email) 的唯一索引，
        否则可能返回 MultipleObjectsReturned。
        """
        try:
            return self.get(**{self.model.USERNAME_FIELD + "__iexact": email})
        except self.model.MultipleObjectsReturned:
            # 记录严重错误，这意味着数据一致性被破坏
            import logging
            logger = logging.getLogger(__name__)
            logger.critical(
                f"Multiple users found for email {email!r}! "
                "Database unique constraint violated!"
            )
            raise
```

## 总结

1. **索引问题**：`iexact` 确实会使标准 B-tree 索引失效，解决方案是创建 `LOWER(email)` 的表达式索引
2. **并发问题**：仅在 Python 层做小写转换存在竞态条件，必须在数据库层添加大小写不敏感的唯一约束
3. **系统侵入性**：对认证系统影响较大（功能上是有益的），但需要注意序列化和管理后台的一致性
